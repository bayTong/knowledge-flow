## SOP-000：知识库最小功能初始化

**定位**：定义「一个能运转的知识库至少需要什么」，是所有后续 SOP 的前置条件。

**触发条件**：用户要求新建知识库，且当前路径下不存在知识库骨架（无 `schema/SCHEMA.md`）。

**前置条件**：
- ☑ 用户已明确知识库覆盖的领域（一个或两个交织领域）
- ☑ 用户已确认浏览工具（如 Obsidian / VS Code / 纯终端）

**两种初始化模式**：

Agent 根据用户是否已有原料，选择对应模式：

| | 模式 A（完整初始化） | 模式 B（骨架初始化） |
|---|---|---|
| 适用场景 | 已有原料，需要立即提炼知识 | 暂无原料，先搭框架等后续填充 |
| 执行步骤 | 全部 9 步（步骤 6 拆为 6a+6b+6c） | 步骤 1-5 + 7-9，步骤 6 标为待完成 |
| 验收标准 | 6/6（或 5/6 当用户在 6b 暂停） | 4/6（⑤⑥ 待原料到达后补验） |
| 关键约束 | 至少一份原料 | 步骤 1（领域确认）不可跳过 |

**输入**：
- 模式 A：领域描述 + 浏览工具选择 + 第一份原料
- 模式 B：领域描述 + 浏览工具选择

**输出**：通过对应模式验收检查的最小可运转知识库

### 步骤 1：确认领域范围

向用户确认以下信息，写入 SCHEMA.md 第一章：

```
本知识库覆盖的领域：
1. [领域 A]——[一句话说明]
2. [领域 B]——[一句话说明]（如有）
```

如果用户无法明确领域，**暂停**——没有领域定义的知识库会退化为混乱的文件堆。等用户想清楚再继续。

### 步骤 2：创建三层目录骨架

在目标路径下创建以下目录结构（空目录用 `.gitkeep` 占位）：

```
<知识库根目录>/
├── raw/
│   ├── articles/          ← 网页文章原文（只读）
│   ├── papers/            ← 论文原文（只读）
│   ├── transcripts/       ← 对话/访谈记录（只读）
│   ├── assets/            ← 图片、图表等附件
│   └── _curation-maps/     ← 粗读器产出的策展地图
├── wiki/
│   ├── entities/          ← 实体页（当前空，待触发）
│   └── queries/           ← 查询存档（当前空，待触发）
├── schema/
└── _archive/
```

> 此时 `wiki/` 下不创建模块子目录。模块编号和命名来自第一份原料的内容归类，不应预先设计。

## 一、领域范围

本知识库覆盖以下领域：

1. **[领域 A]**——[一句话说明]
2. **[领域 B]**——[一句话说明]（如只有一个领域，删除本行）

## 二、目录结构

[三层架构树形图，同步骤 2 创建的结构]

### 模块编号规则

- 两位数字 + 连字符 + 中文模块名，如 `01-核心架构`
- 新增模块时按顺序递增编号
- 模块目录下直接存放页面文件，不设二级子目录

---

## 三、文件命名

### 页面文件
- **使用中文文件名**，以页面标题命名
- 含空格不含连字符，如 `记忆系统四层架构.md`
- 文件名必须与 frontmatter 中 `title` 的主标题部分一致

### 管理文件
- `SCHEMA.md`、`index.md`、`log.md` 使用英文大写名，三者不出现在关系图谱中

### raw/ 原始资料文件
- `articles/`：`作者-主题-年份.md`（如 `karpathy-llm-wiki-2025.md`）
- `papers/`：`作者-年份-简短标题.md`
- `transcripts/`：`YYYY-MM-DD-主题.md`
- raw/ 下所有文件只读不可修改。纠错或更新写入 wiki 页面，不动 raw/

### 禁止
- 英文小写连字符命名
- 文件名与 frontmatter `title` 不一致

---

## 四、Frontmatter 规范

每个页面文件必须以 YAML frontmatter 开头：

---
title: [模块名 · 页面主标题]
created: YYYY-MM-DD
updated: YYYY-MM-DD
type: concept | comparison | entity | query
tags: [来自标签体系]
sources: [信息来源路径或描述]
confidence: high | medium | low
contested: false                 # 可选。true 表示页面存在未解决的矛盾信息
---

> **title 格式按页面类型区分**：
> - `concept` / `comparison`：`模块名 · 页面主标题`
> - `entity`：`实体 · 实体名称`
> - `query`：`问答 · 问题简述`

| 字段           | 必填  | 说明                                                   |
| ------------ | :-: | ---------------------------------------------------- |
| `title`      |  ☑  | 格式见上方按类型区分规则，主标题部分与文件名一致                        |
| `created`    |  ☑  | 页面首次创建日期                                             |
| `updated`    |  ☑  | 最后更新日期，每次修改内容时必须同步更新                                 |
| `type`       |  ☑  | `concept`（机制/原理/架构）、`comparison`（对比分析）、`entity`（工具/模型/人物/组织）、`query`（深度问答存档） |
| `tags`       |  ☑  | 来自标签体系的标签列表，不在体系中的标签不得使用                             |
| `sources`    | 推荐  | 记录信息来源，可为源码路径、URL、或对话描述                              |
| `confidence` | 推荐  | `high`（多来源验证或有源码支撑）、`medium`（单来源或二手信息）、`low`（推测或未验证） |
| `contested`  | 可选  | `true` 表示页面存在未解决的矛盾信息，无矛盾时省略                            |

---

## 五、标签体系

新增标签时必须先在此注册，后使用。

### 主题标签（页面在讲什么）

| 标签 | 含义 |
|------|------|
| [标签1] | [含义说明] |
| [标签2] | [含义说明] |
| [至少填 3 个与领域直接相关的标签] | |

### 性质标签（页面有什么特殊属性）

| 标签 | 含义 |
|------|------|
| `comparison` | 页面包含对比分析（多维表格） |
| `decision` | 页面给出了明确的选择建议或最佳实践 |

> 初始标签只定义当前原料会用到的。原料增加后按需扩充，避免提前设计用不上的标签。

## 六、Wikilink 规范

### 格式

[[文件名|模块名 · 页面标题]]

- 管道符 `|` 前为实际文件名（不含路径、不含 `.md` 扩展名）
- 管道符 `|` 后为完整 frontmatter `title`，用于链接显示文字
- 管道符及显示文字为**必填**

### 交叉引用要求
- 每个页面至少链接 2 个其他页面
- 「相关页面」段落统一放在文件末尾
- 不自链接

---

## 七、页面质量规则

### 创建新页面
- 该主题在当前知识库中尚无页面覆盖
- 或已覆盖但新内容足以构成独立子主题

### 追加到已有页面
- 新信息是对已有内容的补充或纠错，不足以独立成页

### 不创建
- 顺带提及、次要细节、领域外内容

### 拆分
- 单一页面超过约 300 行时考虑拆分为子主题

### 归档
- 内容完全被新版取代 → 移入 `_archive/` → 从 index 移除 → 更新所有指向它的 wikilink

---

## [继续按需补充：八、更新与冲突处理 / 九、导航与日志 / 十、图谱显示 / 十一、操作流程]
```

> **写入后检查**：SCHEMA.md 最少包含 7 个章节（领域范围、目录结构、文件命名、Frontmatter、标签体系、Wikilink、页面质量规则）。

### 步骤 6：执行首次原料摄入

#### 6a：执行 SOP-001（粗读器）——产出策展地图

Agent 以第一份原料为输入，执行 SOP-001「粗读器」完整流程。

SOP-001 将产出：
- raw/ 下的原料文件（含 SHA256 frontmatter）
- raw/_curation-maps/ 下的策展地图（含全景概括、实体清单、关系清单、不确定标记、缺口分析、SCHEMA 建议）
- log.md 追加一条记录：「粗读完成，等待用户审核」

> ⚠ SOP-001 不创建任何 wiki 页面、不更新 index.md。

#### 6b：等待用户审核策展地图

这是一个**人工交互断点**——Agent 不能自动跳过。

用户审核策展地图，确认：
- 哪些实体/概念标记为「确认入库」
- 哪些标记为「忽略」
- 哪些标记为「待更多原料」
- 是否接受 SCHEMA 建议（或提出调整）
- 对不确定标记给出判断

用户在步骤 6b 暂停审核不影响验收——此时 ⑤ 已通过（raw 文件和策展地图均已就位），⑥ 标记为「待完成」。

#### 6c：用户审核通过后，执行 SOP-002（策展入库）

Agent 以审核通过的策展地图为输入，执行 SOP-002「策展入库」完整流程。

SOP-002 将产出：
- wiki/ 下的页面文件（至少一个，含完整 frontmatter 和交叉引用）
- 更新后的 index.md
- 追加的 log.md 条目
- 变更报告 + 自检结果

> **为什么这里调用 SOP-001（粗读器）→ SOP-002（策展入库）而非自己写一套摄入逻辑：** SOP-001 + SOP-002 的组合是知识库增长的唯一入口。第一份原料和第十份原料应走完全相同的两阶段流程（粗读 → 审核 → 策展）。

> **模式 B（骨架初始化）**：如果暂无原料，跳过本步骤全部子步骤。在 log.md 的初始化记录中将「首份原料粗读」标注为「待补充」。

## SOP-001：粗读器（策展地图）

**定位**：所有新原料进入知识库的前置必经环节。Agent 不建库——只产出策展地图，等你审核。

**触发条件**：无条件自动触发。任何新原料到来时，不论 SCHEMA 是否已存在、Agent 对领域是否熟悉，均先走粗读。

触发后，Agent 判断当前 SCHEMA 状态：

| SCHEMA 状态 | 步骤 6 中的附加分析 |
|------------|-------------------|
| 不存在 | 产出完整 SCHEMA 建议（页面类型 + 目录结构 + frontmatter 字段 + 关系类型 + 标签分类法） |
| 存在 | 产出差异分析——新原料中出现但 SCHEMA 未覆盖的类型/关系/标签 |

**你可以随时在粗读前说"跳过粗读，直接策展"——但 Agent 默认走粗读。**

**输入**：一份 raw 原料

**输出**：raw/ 下的原料文件（含 SHA256） + raw/_curation-maps/ 下的策展地图

**硬约束**：

| 编号 | 约束 | 原因 |
|------|------|------|
| C1 | ☒ 不得创建任何 wiki 页面 | 粗读器只产策展地图 |
| C2 | ☒ 每条提取必须有原文引用 | 可追溯 = 可验证 = 可修正 |
| C3 | ☒ 不确定必须显式标记，不得假装确定 | 粗读器区别于策展器的核心价值 |
| C4 | ☒ Agent 建议必须和事实节严格分离 | 防止建议被误认为事实 |
| C5 | ☒ 不得做重要性筛选——穷举提取 | Agent 的遗漏比噪音更难修复 |
| C6 | ☑ 超长原料必须列出"未细读的章节" | 让你知道地图的覆盖边界 |
| C7 | ☑ 隐式关系可以提取但必须标记"推测" + 置信度 ≤ 中 | 不让推测污染确定信息 |

### 步骤 0：捕获原料 + 判断规模和领域归属

#### 0.1 捕获原料到 raw/

将原料存入 `raw/` 对应子目录：

| 原料类型 | 存放路径 | 获取方式 |
|---------|---------|---------|
| URL（静态网页） | `raw/articles/作者-主题-年份.md` | 网页内容提取工具获取 markdown |
| URL（SPA 页面/需 JS 渲染） | `raw/articles/YYYY-MM-DD-主题.md` | **浏览器**加载后提取内容；纯文本提取工具对 SPA 页面无效 |
| 对话记录 | `raw/transcripts/YYYY-MM-DD-主题.md` | 从 LLM 客户端导出或浏览器提取 |
| 论文/PDF | `raw/papers/作者-年份-简短标题.md` | PDF 提取工具处理 |
| 用户粘贴文本 | `raw/articles/` 或 `raw/transcripts/` | 直接写入 |

> **SPA 页面识别与处理**：如果内容提取工具返回的内容明显是摘要/截断/缺少原文结构（如只有标题而无正文段落），说明该页面是 SPA（React/Vue 渲染），纯文本提取工具无法获取完整内容。此时应改用**浏览器**加载页面，通过页面快照或控制台提取全文。如果浏览器也无法获取（如需要登录），标记为 `fidelity=不确定`（见步骤 0.2），在步骤 5 缺口分析中注明"原料完整性无法验证"。

**必须写入 raw frontmatter**：

```yaml
---
source_url: [原始链接]
ingested: YYYY-MM-DD
sha256: [正文的 SHA256 hex digest]
---
[正文内容]
```

SHA256 只对 frontmatter 之后的内容计算。用于后续重复摄入检测——相同 SHA256 则跳过。

#### 0.2 规模分级

| 原料规模 | 策略 |
|---------|------|
| < 500 字（极短） | 全文逐句标注 |
| 500-5000 字（常规） | 逐段标注 + 全景概括 |
| 5000-20000 字（长文） | 先全景概括 → 逐节标注（每节子地图）→ 合并 |
| > 20000 字（超长） | 先全景概括 → 确认重点章节 → 重点逐段标注 → 非重点只概括 → 列出"哪些章节没有细读" |

**超长原料的关键约束：** Agent 必须列出覆盖盲区——让你知道你看到的地图不是全貌。

#### 0.3 多原料同时摄入

Agent 先回答三个问题，和你确认后执行：

> ① 这些原料之间是什么关系？（同一主题补充 / 不同视角 / 不同主题）
> ② 是否应归入同一个知识库？
> ③ 联合粗读还是逐篇粗读？（补充→联合地图 / 不同视角→逐篇+交叉比对 / 不同主题→各自地图）

#### 0.4 轻量领域门禁

粗读前做一次快速判断——原料的核心主题是否**可能**属于当前知识库的领域范围。判断标准从"确定属于"降级为"不显然不属于"：

| 判断 | 处理 |
|------|------|
| ☑ 可能属于 | 继续粗读 |
| ⚠ 不确定（SCHEMA 不存在或 Agent 无法判断） | 继续粗读，步骤 1 全景概括中标注 |
| ☒ 显然不属于 | 提示用户但不终止——用户可以说"我知道，粗读就好" |

### 步骤 1：全景概括

**必须在提取任何实体或关系之前完成。**

Agent 先按原料的自然段落/章节边界，逐段写出内容概括——**每段讲了什么，不遗漏任何一段**。概括的长短由该段的信息密度决定——信息密集的段落可以多写几句，过渡性段落一句话带过。核心约束是"覆盖"，不是"字数"。然后把这些逐段概括拼接为一段连贯的全景叙述。

**逐段概括阶段（必须逐段覆盖，不跳过）：**

```
原文结构                          概括
────────                         ────
第 1 节／段落 1：                这一节定义了 X，引入了 Y 和 Z 两个核心概念…
第 2 节／段落 2：                这一节展开对比了 A 和 B，给出了选型建议…
第 3 节／段落 3：                这一节讨论工程实践，包括…
…（原文有几段就写几条）            …
```

> **如果原文 > 20 段，按节合并（如"第 1-3 段属于引言部分，讲了……"），不在逐段粒度上过度展开。最终拼接后的全景叙述建议控制在 300-600 字，但以"不遗漏原文的主要信息块"为优先，不硬卡字数。**

**拼接整合阶段：**

在逐段概括的基础上，Agent 写一段连贯的全景叙述。回答三个问题：

1. **它讲了什么**（主要）——按原文的逻辑顺序，把逐段概括串联成一个有起承转合的叙述。你读完后能复述出原文的完整信息骨架
2. **它是什么类型的材料**（次要）——一手资料/二手总结/经验分享？信息密度如何？
3. **它和本知识库的关系**（如有 SCHEMA）——领域重叠度？引入了什么 SCHEMA 未覆盖的新方向？

**你读完全景概括后，能判断这篇原料覆盖了哪些主题、各部分之间的关系，以及它在你已有知识体系中的位置。字数不硬限——优先保证信息完整，而非压缩到"快速浏览"。**

---

### 步骤 2：确定提取层次

在深入标注前，Agent 先声明"我将从以下几个层次提取信息"：

```
示例：
本次提取层次：
  - 核心概念（如 RAG、Agent、知识图谱）
  - 技术实体——模型（如 BGE-M3）
  - 技术实体——工具/框架（如 Chroma）
  - 事实主张（如"MRR提升15-40%"）
  - 关系（实体之间的显式连接）

选层依据：原文以"概念解释 + 工具选型对比 + 质量数据"为主线。
```

**首篇文章冷启动时：** Agent 自行归纳层次——你的审核决定这些层次是否合适，合适的层次后续固化为 SCHEMA 的一部分。

**后续文章：** Agent 以已有 SCHEMA 的类型体系为基准，新增的层次显式标注"新增层次"。

---

### 步骤 3：逐段标注——穷举提取

**提取接口规范（每一条提取必须包含以下 7 个字段）：**

| 字段 | 含义 | 示例 |
|------|------|------|
| `标识符` | 唯一 ID | `@concept:rag` |
| `名称` | 人类可读 | `RAG（检索增强生成）` |
| `所属层次` | 在步骤 2 确定的层次中选一个 | `核心概念` |
| `一句话描述` | 原文中如何定义 | `检索→增强→生成，扩展模型记忆` |
| `原文引用` | 精确到段落 | `第 1 节第 1 条` |
| `置信度` | 确定 / 推测 / 不确定 | `确定` |
| `不确定原因` | 仅置信度 ≠ 确定时填写 | `—`（见下方 5 种类别） |

**不确定标记的 5 种原因类别：**

| 类别 | 含义 | 典型场景 |
|------|------|---------|
| `来源不可靠` | 数值/声明没有原始引用 | "MRR 提升 15-40%"，无论文引用 |
| `Agent 理解局限` | Agent 不确定自己理解是否正确 | 语义模糊的段落 |
| `原文模糊` | 原文本身的表述不够明确 | 类比过于简化 |
| `信息不完整` | 原文提到了但未展开 | 提到 Graph RAG 但没给实现方案 |
| `总结压缩损失` | 原料本身是二手总结 | 对话总结而非原始记录 |

> **多 Pass 分治实现**：步骤 3–5（逐段标注、关系提取、缺口分析）融合了三个维度的认知任务——实体识别、关系判断、论点提取。一次性执行可能导致注意力分散和遗漏。`prompts/` 目录提供了三 Pass + 组装的分治实现：Pass 1 负责步骤 1–3（全景概括 + 提取层次 + 实体清单），Pass 2 负责步骤 4（关系清单），Pass 3 负责步骤 5（事实主张），最后由组装模板补全缺口分析、SCHEMA 建议和 Agent 签名。详见 `prompts/README.md`。

---

### 步骤 4：关系提取

**显式关系：** 原文明确说的——高置信度，必须提取。

**隐式关系：** Agent 推断的——可以提取但必须：
- 标记为"推测"
- 标注推测依据（"因为 A 和 B 在原文中并列讨论……"）
- 置信度不得设为"确定"

---

### 步骤 5：缺口分析

Agent 列出"如果这个知识库的目标是 X，那么这篇原料应该涵盖但未涵盖的内容"。

**如果目标不明：** Agent 在缺口分析前先问你——

> "这篇原料你想从中获得什么？——
> ① 理解概念框架 ② 为工程实践做技术选型参考 ③ 建立知识库的结构化索引 ④ 其他（请说明）"

如果你回答"对领域不熟，说不上来"——Agent 按默认目标（① 理解概念框架）执行缺口分析，并在缺口分析开头标注。

---

### 步骤 6：SCHEMA 建议

**情况 A——SCHEMA 不存在（首篇文章）：** Agent 从地图中推导完整 SCHEMA 建议，包括页面类型（名称 + 对应目录 + frontmatter 字段）、关系类型（边类型 + 含义 + 示例）、标签分类法。

**情况 B——SCHEMA 已存在：** Agent 列出差异分析：

```
现有 SCHEMA 已覆盖：[已有类型]
新原料引入但未覆盖：
  + [新类型]（新，建议增补）
  + [新关系类型]（新，建议增补）
```

---

### 步骤 7：Agent 建议（严格隔离）

**格式：** 每个提取节之后，用独立子标题 `### Agent 建议（非事实，仅供参考）`。

**允许的内容：**
- 关联你已有的知识或工具
- 提醒你可能重点关注的部分
- 发现原料内容和你已有 wiki 的潜在关联
- 标注排序建议

**禁止的内容：**
- 原文中没有的信息
- "你应该做什么"的指令性建议
- 以确定语气表述的推断

---

### 步骤 8：产出策展地图

保存至 `raw/_curation-maps/[YYYY-MM-DD]-[主题].md`。

**地图结构（9 节）：**

```
1. 全景概括             ← 逐段缩写 + 拼接整合，讲清楚"原文说了什么"
2. 提取层次说明          ← Agent 将从哪几层提取
3. 核心概念 / 实体清单    ← 分组表格，每条 7 字段
4. 关系清单              ← 区分显式/隐式
5. 事实主张              ← 数值/声明 + 置信度
6. 不确定标记             ← 需要你重点核实的内容
7. 缺口分析              ← 原料应涵盖但未涵盖的内容
8. SCHEMA 建议           ← 完整方案 or 差异分析
9. Agent 签名            ← 提取统计 + "未创建 wiki 页面，等待审核"
```

**本 SOP 结束后：** Agent 等待你审核地图。不自检格式、不建 wiki 页面、不更新 index。

## SOP-002：策展入库

**定位**：以审核通过的策展地图为输入，在 SCHEMA 约束下精确执行 wiki 页面创建/更新。摄入管道的第二段——粗读器完成了"提取"和"你审核"之后，这里做的是"写入"。

**触发条件**：
- 用户审核策展地图后说"确认入库"/"可以建库"
- SOP-006（话题提炼）用户选"入库"

**输入**：审核通过的策展地图 + 当前知识库完整状态（SCHEMA + index + 已有页面）

**输出**：新建/更新的 wiki 页面 + 更新的 index.md + 追加的 log.md + 变更报告

**前置**：
- ☑ 粗读器已产出策展地图，且用户已审核通过（标记了哪些"确认入库"）
- ☑ raw/ 下存在对应的原料文件，且 `source_url` + `ingested` + `sha256` 三个 frontmatter 字段齐全（由 SOP-001 步骤 0.1 保证）
- ☑ SCHEMA.md 存在且可读取
- ☑ index.md 和 log.md 存在
- ☑ raw/ 和 wiki/ 目录存在

**总览**：

```
步骤 1：验证前置 + 读取审核通过的地图 + SCHEMA
       · 确认 raw 原料文件存在且 SHA256 有效
       · 确认策展地图中用户已标记"确认入库"的条目
步骤 2：定向——检查已有 wiki 页面，防重复
...
步骤 3：决策——仅处理地图中标记为"确认入库"的条目
       · 创建 / 追加 / 标记矛盾 / 跳过
       · 确定归属模块
步骤 4：SCHEMA 同步检查（新标签/新模块/领域扩展）
步骤 5：按模板写入 wiki 页面
       · concept / comparison / entity / query
       · 8 条写入通用约束
步骤 6：更新导航——index.md + log.md
步骤 7：自检（8 项）
步骤 8：报告——变更清单 + 待关注标记
```

---

### 步骤 1：验证前置 + 读取审核通过的地图 + SCHEMA

**1.1 验证 raw 原料文件存在且 SHA256 有效：**

确认 SOP-001 步骤 0.1 捕获的原料文件在 `raw/` 对应子目录下。打开文件，验证 frontmatter：

```yaml
---
source_url: [原始链接，无则写来源描述]
ingested: YYYY-MM-DD
sha256: [64 位 hex digest]
---
[正文内容]
```

- `source_url` 非空
- `ingested` 日期格式正确
- `sha256` 为 64 位 hex 字符串且非空

> **SHA256 计算规则：** 只对 frontmatter 之后的内容（第二个 `---` 之后）取 SHA256。不可对整个文件取——frontmatter 中的 `ingested` 日期每次不同，会导致同一份原料算出不同哈希。如果 SHA256 缺失或为占位符，说明 SOP-001 步骤 0.1 执行不完整——暂停，回到 SOP-001 补完捕获步骤，重新计算 SHA256 后写入 raw frontmatter。

> **重复摄入检测：** 如果 `raw/` 中已存在相同 `source_url` 的文件，对比 SHA256。相同则原料未变化，不需要策展入库——终止，报告用户「该原料已摄入且未变化」。SHA256 不同则原料已更新，继续策展流程，步骤 8 报告中注明「来源已更新」。

> **JSONL 例外：** 如果原料为 `.jsonl` 文件（对话导出），写一个同名 `.md` 配套文件存放 frontmatter，SHA256 对 JSONL 文件计算。

**1.2 确认策展地图审核状态：** 打开 `raw/_curation-maps/` 下对应的策展地图，确认其中存在用户审核标记——至少有一个条目标记为「确认入库」。如果地图未审核（无任何标记），暂停，提示用户先审核。

**1.3 提取确认入库的条目：** 从策展地图中提取所有标记为「确认入库」的实体、概念、关系、事实主张。这些是后续步骤的操作对象——只处理这些，不处理标记为「忽略」或「待更多原料」的条目。

**1.4 确认 SCHEMA 当前版本：** 如果用户在审核地图时调整了 SCHEMA（如新增标签、调整页面类型），确认 SCHEMA.md 已更新且与用户的调整一致。

---

### 步骤 2：定向——理解知识库现状

在写入任何内容之前，必须先理解当前状态。这是防止重复创建、遗漏关联的核心步骤。

**2.1 读 SCHEMA.md**：确认当前规范——标签体系是否有更新、命名规则是否有调整、Wikilink 格式是否一致。这是后续所有写入操作的约束来源。

**2.2 读 index.md**：获取已有页面的完整清单。对每个模块下的页面建立心智模型——哪些概念已覆盖、哪些还空着。index 的摘要只有一句话，无法覆盖页面内的全部子主题，因此**必须追加全文搜索**。

**2.3 全文搜索（不可跳过）**：对地图中每个「确认入库」的条目，取核心关键词，在 `wiki/` 目录下所有 `.md` 文件中全文搜索关键词。

> 使用 LLM 客户端提供的文件搜索功能，或用 `grep` 等效命令行工具。搜索范围限定 `wiki/` 目录。

> **为什么不可跳过**：index.md 的摘要只有一句话，无法覆盖页面内的全部子主题。一份原料中看似新的内容，可能在某个已有页面的深层章节中已经写过。跳过全文搜索直接创建新页面，会导致两份独立页面讲同一件事——这是知识库最致命的退化模式。

**2.4 输出定向结论**：完成上述三步后，Agent 应能回答：

```
确认入库的条目中：
  - 已有一致页面：[列出]        → 走「追加」
  - 已有近似但需重写：[列出]    → 走「更新 + 标记」
  - 尚无覆盖：[列出]            → 走「创建」
  - 已有页面已完全过时：[列出]  → 走「更新 + 触发 SOP-005 归档旧版」
```

定向结论决定了步骤 3 决策树中每一条的分支走向。如果定向阶段遗漏了已有页面，后续步骤会创建重复页面——因此全文搜索是定向阶段最关键的步骤。

---

### 步骤 2.5：领域门禁

**判断依据**：SCHEMA 第一章定义的领域范围。不看关键词，看主题本质。

**三级出口**：

| 判断 | 处理 |
|------|------|
| ☑ 完全在领域内 | 继续步骤 3 |
| ⚠ 部分在领域内 | 自动提取领域内部分。报告中注明「部分内容未摄入（领域外）」 |
| ☒ 完全在领域外 | 触发人工介入 |

**☒ 出口的人工介入流程**：

```
⚠ 原料主题与知识库领域存在显著差异

   原料主题：[一句话概括]
   知识库领域：[引用 SCHEMA 第一章]
   差异说明：[具体理由]

   选项 1：仍然摄入（将自动扩展 SCHEMA 第一章领域范围）
   选项 2：跳过
   选项 3：创建独立知识库
```

用户选选项 1 时，Agent 自动更新 SCHEMA 第一章，追加新领域描述。报告中注明「领域范围已扩展」。

---

### 步骤 3：决策——门禁检查核心

对步骤 2 定向后的每一项：

```
对于地图中每个「确认入库」的条目：

① 知识库里是否已有页面覆盖这个主题？
   │
   ├── 是 → 新信息与已有页面相比：
   │        ├── 是对已有内容的补充或细化
   │        │   → 追加到已有页面
   │        │
   │        ├── 与已有内容完全一致（无新增事实、无新观点）
   │        │   → 仅在 sources 中追加原料路径，不修改正文
   │        │   → 报告标注为「已覆盖（来源 +1）」
   │        │
   │        ├── 与已有内容矛盾
   │        │   → 追加 + 标记矛盾：frontmatter contested: true
   │        │       正文中注明新旧双方观点和各自来源日期
   │        │
   │        └── 已有页面已完全过时，新信息全面覆盖
   │            → 更新页面，旧版内容可能触发 SOP-005 归档
   │
   └── 否 → 创建新页面
            确定页面类型（concept / comparison / entity / query）
            确定归属模块（已有模块 or 新建模块 编号递增）
```

**创建 entity 页面的特殊规则**：至少被 1 个 concept 页面通过 wikilink 引用，否则不创建。存放于 `wiki/entities/`。

**创建 query 页面的特殊规则**：回答内容有独立保留价值、不能简单追加到已有 concept 页面。存放于 `wiki/queries/`。

**确定归属模块**：新页面属于哪个模块，不由原料本身决定，看页面内容的主题本质：

```
1. 列出当前 wiki/ 下已有模块及各自覆盖的主题范围
2. 新页面的核心主题与哪个已有模块最接近？
   ├── 找到 → 归入该模块
   └── 找不到 → 新建模块（编号递增）
```

> 判断依据是 SCHEMA 第一章定义的领域方向和已有模块的实际内容。

> **与粗读器的分工：** 粗读器穷举提取——不在乎是否已有页面。策展入库只处理你确认过的条目——并在决策树中执行查重和防冲突。

### 步骤 4：SCHEMA 同步检查

在写入页面之前，确保本次摄入需要的新标签和新模块已在 SCHEMA 中注册。

**检查 1：新标签注册**

本次决策需要使用的标签中，是否有未在 SCHEMA 第五章注册的？

- 已在 → 继续
- 未在 → Agent 自动注册：在 SCHEMA 第五章「主题标签」表中新增一行。步骤 8 报告中注明：「新增标签：[标签名]」

**检查 2：新模块注册**

本次创建的页面是否属于新模块？

- 已有模块 → 继续
- 新模块 → Agent 自动更新 SCHEMA 第二章目录树。步骤 8 报告中注明：「新增模块：[编号-名称]」

**检查 3：领域扩展确认**

步骤 2.5 是否扩展了领域范围？如果是，确认 SCHEMA 第一章已更新。

**检查 1 或 2 触发了 SCHEMA 修改时**：步骤 8 报告中注明「已修改 SCHEMA」，SOP-002 结束后自动执行 SOP-004。

### 步骤 5：按模板写入 wiki 页面

```

---
title: 实体 · 实体名称
created: YYYY-MM-DD

type: entity
tags: [标签1, 标签2]
sources: [raw/...]

|------|-----|

```

> entity 存放于 `wiki/entities/`；concept 存放于 `wiki/0N-模块名/`。entity 回答「X 是什么（属性）」，concept 回答「X 怎么工作（机制）」。

##### 5.4 Query 页面模板

```markdown
---
title: 问答 · 问题简述
created: YYYY-MM-DD

## 相关页面
```

> query 默认 `confidence: medium`，除非有 2+ 来源交叉验证。

##### 5.5 写入通用约束（8 条）

| # | 约束 | 违反后果 |
|---|------|---------|
| ① | `title` 格式为 `模块名 · 页面主标题`，主标题部分与文件名一致 | 图谱节点显示不一致 |
| ② | `tags` 全部来自 SCHEMA 标签体系 | 标签审计不通过（SOP-003 会捕获） |
| ③ | `sources` 字段指向步骤 0.1 存入的 raw 文件路径 | 溯源链断裂 |
| ④ | 页面至少包含 2 个 wikilink（entity 至少 1 个入站链接） | 孤立页面（SOP-003 会捕获） |
| ⑤ | wikilink 格式为 `[[文件名\|模块·标题]]` | 灰色节点 |
| ⑥ | 「相关页面」段落放在文件末尾 | 结构不统一 |
| ⑦ | 不自链接 | 图谱自环 |
| ⑧ | `updated` 日期必须是实际修改日期 | 过期检测失效 |

**追加到已有页面的额外要求**：在正文中插入新内容（非简单末尾追加），更新 `updated`，追加 `sources`。如果导致页面超 300 行，步骤 8 报告中标记为「拆分候选」。

### 步骤 7：自检（8 项）

逐项检查本次摄入涉及的全部页面（新建 + 更新）。任意一条未通过则修复后重检。

| #   | 检查项            | 通过标准                                                                   | 范围      |
| --- | -------------- | ---------------------------------------------------------------------- | ------- |
| ①   | 文件名与 title 一致  | 文件名 = title 中「· 」后面的主标题部分                                              | 新建页面    |
| ②   | Frontmatter 完整 | `title` `created` `updated` `type` `tags` `sources` `confidence` 七字段齐全 | 新建 + 更新 |
| ③   | 标签在体系内         | 所有 `tags` 值均在 SCHEMA 标签体系中注册过                                          | 新建页面    |
| ④   | Wikilink 格式正确  | 使用 `[[文件名\|模块·标题]]` 管道格式                                               | 新建 + 更新 |
| ⑤   | 至少 2 个出站链接     | 页面至少链向 2 个其他页面（entity 除外，但须有 ≥1 个入站链接）                                 | 新建页面    |
| ⑥   | 无自链接           | 页面不在自己的「相关页面」中引用自己                                                     | 新建 + 更新 |
| ⑦   | 入站回链           | 新增了 wikilink 指向某个已有页面 → 检查那个页面是否也该回链                                   | 新建页面    |
| ⑧   | SHA256 已写入     | raw 文件的 frontmatter 含 `sha256` 字段且值非空（由 SOP-001 步骤 0.1 保证）             | raw 文件  |

> 如果自检失败超过 3 项：检查步骤 5 写入时是否严格遵循了模板。

## SOP-003：全量健康扫描（Lint）

**定位**：知识库的定期健康检查。两次 SOP-002 之间知识库在无人看守状态下发生的变化，是 SOP-003 要捕获的目标。

**触发方式**：

| 触发 | 命令 | 范围 | 说明 |
|------|------|------|------|
| 手动 | 用户说「跑 Lint」 | 全量 9 项 | 立即执行，输出完整报告 |
| 定时 | 通过 LLM 客户端的定时任务功能设置，建议每周一次 | 全量 9 项 | 后台执行，结果推送用户。全部通过可静默 |
| 摄入后自动 | 每次 SOP-002 完成后自动触发 | 轻量 3 项 | 仅查断裂 wikilink / 孤立页面 / entity 孤立。SOP-002 修改了 SCHEMA 时由 SOP-004 替代（断裂/孤立由下次定时全量版覆盖） |

**原则**：定位问题 → 给出原因与解决方案 → 等人决定。Agent 不自行修复，除非以下两项：

| 自动修复 | 原因 |
|---------|------|
| 日志轮转 | 纯机械操作，可逆 |
| index 缺条目补全 | 格式固定，Agent 读页面写摘要可自动完成 |

---

### 检查项 1：断裂 wikilink

**做什么**：扫描 `wiki/` 下所有 `.md` 文件中所有 `[[...|...]]`，提取管道符前的文件名，检查对应文件是否存在。

**为什么 SOP-002 自检发现不了**：SOP-002 自检只查格式，不查目标文件是否存在。目标文件可能在摄入之间被归档或重命名。

**输出**：

```
[错误] 断裂 wikilink（N 条）
  · [[xxx|标题]] → 文件 xxx.md 不存在
    所在页面：wiki/模块/页面.md
    原因可能：目标文件被归档、重命名、或从未创建
    建议：确认目标文件当前正确名称，修正 wikilink 中的文件名部分
```

> **实现说明**：扫描时须排除行内代码（`` `...` ``）和代码块（` ```...``` `）内的 `[[xxx]]`。这些是代码示例，Obsidian 不会将其渲染为 wikilink。

---

### 检查项 2：孤立页面

**做什么**：对 `wiki/` 下每个页面，统计其他页面中引用它的 wikilink 数量。入站链接为 0 = 孤立。

> entity 页面入站链接为 0 单独归类为**检查项 8**，不在此处报告。

**为什么 SOP-002 自检发现不了**：新建页面刚创建时入站链接可能已在 SOP-002 步骤 7 第 ⑦ 条回链到位。但引用它的那个页面如果后来被归档了，入站链接重新归零——SOP-002 不再检查未被更新的旧页面。

**输出**：

```
[错误] 孤立页面（N 页）
  · wiki/模块/页面.md — 入站链接 0，其余页面均未引用它
    原因可能：引用它的页面被归档、或从未建立回链
    建议：在相关页面的「相关页面」段新增 [[页面名|标题]]，
    或确认该页面是否应归档
```

---

### 检查项 3：index 完整性

**做什么**：列出 `wiki/` 下所有 `.md` 文件，逐一对比是否在 index.md 中有对应条目。

**输出**：

```
[警告] index 缺失条目（N 条）— 已自动修复
  修改前：index.md 中无 [页面名] 条目
  修改为：在对应分组下新增：
          [[文件名|标题]] — [从页面首段提取的摘要]
  回滚方式：删除 index.md 中上述新增行

[警告] index 多余条目（N 条）
  · index.md 中「[[xxx|标题]]」但文件 wiki/xxx.md 不存在
    原因可能：文件被归档但 index 未同步更新
    建议：删除该条目
```

> **自动修复仅针对缺失条目**。多余条目（文件已删但 index 残留）Agent 不自动删除——无法判断是人类有意保留的占位还是遗漏。

---

### 检查项 4：Frontmatter 校验

**做什么**：检查每个页面是否包含全部 7 个必填/推荐字段，`type` 值是否在 4 种之内，`title` 格式是否与页面类型匹配。

**输出**：

```
[错误] Frontmatter 缺字段（N 页）
  · wiki/模块/页面.md — 缺少 updated
    建议：补充 updated: YYYY-MM-DD（最后修改日期）

[警告] type 值不合法（N 页）
  · wiki/模块/页面.md — type: note（不在 SCHEMA 允许的 4 种类型内）
    建议：改为 concept / comparison / entity / query 之一
```

---

### 检查项 5：标签审计

**做什么**：收集所有页面 `tags` 字段中的全部标签，与 SCHEMA 第五章注册列表对比。标记未注册标签和被注册但未使用的标签。

**输出**：

```
[警告] 未注册标签（N 个）
  · 标签「sop」出现在 wiki/模块/页面.md，但未在 SCHEMA 第五章注册
    建议：如该标签确需使用，先在 SCHEMA 第五章注册；
    若为拼写错误，修正页面中的 tags

[提醒] 已注册但未使用（N 个）
  · SCHEMA 中注册的标签「caching」当前无任何页面使用
    建议：从 SCHEMA 第五章移除或保留供未来使用
```

---

### 检查项 6：页面过大

**做什么**：统计每个页面行数，超过 300 行的标记为拆分候选。entity 页面不拆分，query 页面通常不拆分。

**输出**：

```
[警告] 页面过大（N 页）
  · wiki/模块/页面.md — 350 行（阈值 300）
    建议：拆分为 [子主题 A] + [子主题 B] 两个独立页面
```

---

### 检查项 7：日志轮转

**做什么**：检查 log.md 行数。超过 500 条 → 自动轮转。

**输出**：

```
[已自动修复] 日志轮转
  修改前：log.md 共 523 条，最近条目 [日期]
  修改为：log.md → log-YYYY.md（归档），新建空白 log.md
  新 log.md 首行：## [YYYY-MM-DD] create | 日志轮转自 log-YYYY.md
  回滚方式：删除新 log.md，将 log-YYYY.md 重命名回 log.md
```

> 未达 500 条则跳过，不在报告中出现。

---

### 检查项 8：entity 孤立

**做什么**：对 `wiki/entities/` 下的每个 entity 页面，检查是否有 concept 页面引用它。入站链接为 0 = entity 孤立。

**与检查项 2 的区别**：entity 的交叉引用是入站驱动——「有 concept 链它」比「它链别人」更重要。因此 entity 孤立单独归类，不与 concept 孤立混在一起。

**输出**：

```
[错误] entity 孤立（N 个）
  · wiki/entities/DeepSeek V4 Pro.md — 无 concept 页面引用
    原因可能：引用它的 concept 页面被归档
    建议：在相关 concept 页面的「相关页面」中新增 wikilink，
    或考虑归档该 entity 页面
```

---

### 检查项 9：图谱过滤规则

**做什么**：检查 `.obsidian/graph.json` 是否存在且 `search` 字段包含 `-path:schema`。

**为什么 SOP-002 自检发现不了**：Obsidian 关闭时会自动覆写 `graph.json`，可能清空手工设置的 `search` 过滤规则——这是已知陷阱。SOP-000 步骤 8 初始化时配置正确，但后续 Obsidian 运行期间可能被静默改写。

**输出**：

```
[错误] 图谱过滤缺失
  · .obsidian/graph.json 中 search 字段为空
  → SCHEMA.md、index.md、log.md 出现在图谱中，index.md 成为虚假 hub 节点
    修复：将 "search": "" 改为 "search": "-path:schema"
    回滚方式：将 search 改回空字符串
```

> **可自动修复**：补全 `-path:schema` 到 search 字段，纯机械操作，可逆。自动修复后报告中注明修改前状态和回滚方式。

### 轻量版（摄入后自动触发）

## SOP-004：SCHEMA 一致性检查

**定位**：SCHEMA 被修改后，检查修改有没有造成连锁破坏——SCHEMA 内部是否自洽、SCHEMA 与实际文件是否一致。

**触发条件**：

| 触发 | 场景 |
|------|------|
| SOP-002 改动了 SCHEMA | 步骤 2.5 扩展了领域范围、步骤 4 注册了新标签或新模块。SOP-002 结束后立即执行 |
| 用户指令 | 用户明确说「改 SCHEMA」、「注册标签」、「调整规范」等。修改完成后立即执行 |
| SOP-003 发现不一致 → 用户确认修 | SOP-003 报告了 SCHEMA 和页面之间的结构性差异，用户说「同步 SCHEMA」。Agent 改完 SCHEMA 后执行 |

**原则**：定位影响 → 给出建议 → 等人决定。Agent 不自行修改 SCHEMA 或页面，除非图谱色组补全。

| 自动修复 | 原因 |
|---------|------|
| 图谱色组补全 | 多一个模块 → 加一种颜色，纯机械操作，可逆 |

---

### 检查项 1：领域 ↔ 标签

**做什么**：SCHEMA 第一章新增或修改了领域描述 → 检查第五章标签体系是否需要预留对应标签。

**输出**：

```
[建议] 新领域 → 标签预留
  修改：SCHEMA 第一章新增领域「[新增领域描述]」
  第五章现状：无对应标签
  建议新增标签：[标签名] — [含义]
  操作：等待用户确认后注册
```

> 不检查「删了领域要不要删标签」——删除标签是破坏性操作，需用户明确指令。

---

### 检查项 2：目录结构 ↔ 实际文件

**做什么**：SCHEMA 第二章描述的目录结构和 `wiki/` 下实际存在的目录是否一致。

- 规则超前（SCHEMA 写了但目录不存在）→ 报告
- 规则滞后（目录存在但 SCHEMA 没写）→ 报告

**输出**：

```
[警告] 目录结构不一致
  规则超前：
  · SCHEMA 写「wiki/0N-模块/」但目录不存在
    → 如模块尚未创建，忽略；如应已存在，手动创建

  规则滞后：
  · 目录「wiki/0N-模块/」存在但 SCHEMA 第二章未列出
    → 建议更新 SCHEMA 第二章目录树
```

---

### 检查项 3：标签体系 ↔ 页面使用

**做什么**：SCHEMA 第五章新增了标签 → 列出已在用该标签的页面（之前的 SOP-003「未注册」警告可解除）。SCHEMA 计划删除标签 → 列出仍在用该标签的页面（将变成违规）。

**输出**：

```
[需要关注] 标签变更影响
  新增标签：
  · SCHEMA 新增「[标签名]」
  · 已有 N 个页面使用此标签：[列表]
  · 下次 SOP-003 中「未注册标签」警告将自动解除

  删除标签：
  · SCHEMA 计划删除「[标签名]」
  · 仍有 N 个页面使用此标签：[列表]
  · 建议：先更新这些页面的 tags，再删除标签
```

---

### 检查项 4：Wikilink 规范 ↔ 现有页面

**做什么**：SCHEMA 第六章修改了 wikilink 格式规则 → 列出受影响页面及数量。

**输出**：

```

  修改：[描述 SCHEMA 第六章的具体变更]
  影响：N 个页面共 M 条 wikilink 使用旧格式
  列表：
  · wiki/模块/页面.md — N 条
  · ...
  建议：逐页修改或保持向后兼容
  操作：等待用户确认
```

---

### 检查项 5：图谱配置 ↔ 模块数量

**做什么**：`.obsidian/graph.json` 中颜色组数量是否匹配 `wiki/` 下实际模块数量。

**输出**：

```

  → 已自动修复：为 0N-模块名 新增色组
     回滚方式：删除 graph.json 中新增的 colorGroups 条目

[警告] 图谱色组多余
  · graph.json 含 0N-模块，但 wiki/ 下无此目录
    建议：删除多余的 colorGroups 条目
```

> 自动修复仅补全缺失色组。多余的只报告不自动删。

## SOP-005：页面归档

**定位**：wiki 页面被新版本取代或确认无用后，退役到 `_archive/`，清理残留引用。只归档 wiki 页面——raw/ 不可变，schema/ 治理文件不归档。

**触发条件**：

| 场景 | 入口 |
|------|------|
| 摄入时旧页面被全面覆盖 | SOP-002 步骤 3 的「完全过时」出口 → 触发本 SOP |
| Lint 发现孤立/无用页面，用户决定归档 | SOP-003 报告 → 用户说「归档」 |
| 用户手动指令 | 用户说「归档 [页面名]」 |

---

### 步骤 1：确认归档目标

向用户确认：

```
确认归档：
  页面：[路径]
  原因：[被 [新页面] 取代 / SOP-003 标记孤立 / 用户手动]
  替代页面：[有则写路径，无则写「无」]
```

用户确认后执行。SOP-002 触发时默认已确认（摄入报告已展示替代关系），不需重复确认。

---

### 步骤 2：移动文件到 _archive/

保留原始目录结构：

```
wiki/01-核心架构/旧页面.md → _archive/01-核心架构/旧页面.md
wiki/entities/旧实体.md     → _archive/entities/旧实体.md
wiki/queries/旧问答.md      → _archive/queries/旧问答.md
```

目标目录不存在则创建。

---

### 步骤 3：从 index.md 移除条目

删除 index.md 中对应行。如有替代页面，确认替代页面已在 index 中。

---

### 步骤 4：扫描残留链接

扫描 `wiki/` 下所有 `.md` 文件，找出指向被归档页面的 wikilink。按场景分类：

```
残留链接扫描 — [被归档页面]

  有替代页面（N 条）：
  · wiki/模块/A.md → [[旧页面|...]]，可替换为 [[新页面|...]]

  无替代页面（N 条）：
  · wiki/模块/B.md → [[旧页面|...]]，无替代目标

  自引用：
  · 被归档页面自身的「相关页面」段落 → 随页面一起归档，忽略
```

---

### 步骤 5：报告用户，等待决策

```
归档报告 — [页面名]

  已执行：
  · 文件已移至 _archive/模块/页面.md
  · index.md 条目已移除

  待处理残留链接（N 条）：

  [替代可用]
  · wiki/模块/A.md → [[旧页面|标题]]
    建议：替换为 [[新页面|标题]]

  [无替代]
  · wiki/模块/B.md → [[旧页面|标题]]
    建议：移除该 wikilink

  请确认上述处理方案，或逐条指定。
```

用户确认后执行步骤 6。

---

### 步骤 6：执行链接处理

按用户确认的方案逐条修改。完成后追加 log：

```
## [YYYY-MM-DD HH:MM:SS] archive | [页面名]

- 归档原因：[被取代 / 孤立 / 用户手动]
- 文件：wiki/模块/页面.md → _archive/模块/页面.md
- 替代页面：[有则写]
- 残留链接：N 条（已处理 N 条）
- index.md：条目已移除
```

---

### 与 SOP-002 的衔接

SOP-002 步骤 3 决策树的「完全过时」出口：

- 如果新信息全面覆盖了已有页面 → 触发 SOP-005「页面归档」
- 如果只是部分取代，不适合整体归档 → 保留为页面内「历史版本」段落

### 与其他 SOP 的关系

|         | 关系                                                     |
| ------- | ------------------------------------------------------ |
| SOP-002 | 步骤 3 触发——摄入发现旧页面被全面覆盖                                  |
| SOP-003 | 发现孤立页面 → 用户决定归档 → 触发本 SOP。归档后残留链接如果遗漏，下次 SOP-003 会发现断裂 |
| SOP-004 | 不直接相关——归档不改 SCHEMA                                     |

## SOP-006：话题切换提炼提案

**定位**：对话中 Agent 检测到上一个话题可能已结束、用户转向新话题时，自动评估并生成提炼提案，由用户决定是否入库。

SOP-006 只做三件事：检测 → 提案 → 等待决策。不导出对话、不修改文件、不执行摄入。用户决策「入库」后交由 SOP-002 执行。

**触发条件**（两条全部满足）：

```
条件 1：上一个话题持续了 ≥ 8 轮
  → 浅尝辄止的话题不值得提炼

条件 2：当前话题与上一个话题无直接关联
  判断方式：新话题的讨论对象与上一个话题不存在引用、延伸或从属关系
  例：聊完 A 机制切换到 B 配置 → 触发（无关）
  例：聊完 A 机制切换到 A 的缓存 → 不触发（延伸）
  例：讨论 B 时引用 A 作例子 → 不触发（引用）
```

**预判**：触发后两条都通过才生成提案，否则静默：

```
预判 1：上一段对话产出了可提炼的内容
  → 不是纯闲聊、纯调试、纯 yes/no 确认

预判 2：预判未能排除 → 生成提炼提案
```

**提案格式**：

```
─ 提炼提案 · 上一个话题 ─

  主题：[一句话概括]
  建议摄入：
  · 新建：[页面名称] — [内容概述]
  · 更新：[已有页面] — [追加/修改什么]
  领域：[属于哪个已有领域 / 建议扩展领域 / 建议新建知识库]

  ─
  [入库] · [不存] · [修改意见]
```

**用户决策**：

| 选择   | Agent 行为 |
| ---- | ---- |
| 入库 | 执行以下子步骤后，调用 SOP-002（策展入库）：① Agent 导出对话到 `raw/transcripts/YYYY-MM-DD-主题.jsonl`；② 对导出文件计算 SHA256，写入配套 `.md` frontmatter（含 `source_url`、`ingested`、`sha256`）；③ 基于提炼提案 + 对话原文，生成**简化版策展地图**——仅包含提案中建议入库的条目，标记 `confidence: high`（你刚参与对话，信息可信度高），不确定标记留空或标为"实时审核"；④ 简化地图存入 `raw/_curation-maps/YYYY-MM-DD-主题.md`；⑤ 调用 SOP-002 完整流程。SOP-002 从简化地图中读取确认入库的条目——与从 SOP-001 完整地图中读取的路径完全一致 |
| 不存 | 跳过 |
| 修改意见 | 调整后，按上述「入库」子步骤执行 |

> **为什么是"简化地图"而非完整粗读**：SOP-006 的场景下，你刚刚和 Agent 讨论了 8+ 轮——你已经实时审核了对话内容。粗读器的核心价值是帮你在"面对陌生原料时"获得判断力，而你在这里不需要——你已经有了判断力。简化地图的作用是把你的判断（提炼提案中的"入库"选择）转换为 SOP-002 能消费的标准格式。

**入库后**：SOP-002 完整流程 → 改 SCHEMA 则 SOP-004 / 未改则 SOP-003 轻量版。

## 附录 A：批量摄入

一次摄入**多份原料**时（由 SOP-001 步骤 0.3 判断归属后）：

```
1. 全部原料执行 SOP-001（各自产出策展地图 or 联合产出联合地图）
2. 地图统一提交你审核
3. 审核通过后，合并执行 SOP-002：
   - 跨原料查重：原料 B 的实体可能已被原料 A 覆盖
   - 统一写入 wiki 页面 + 一次性更新 index 和 log
4. SOP-002 自检覆盖全部产物
```

关键：步骤 3 的跨原料查重是批量模式的核心——避免多份原料分别创建了同一个概念的两个页面。
