# 自适应提取分层 —— 修改方案

> 审核日期：2026-07-13
> 当前问题：prompts/ 已实现 3 Pass + 组装 = 4 次 LLM 提取 + 1 次策展 = 5 次/篇。一刀切多轮次，短文也走重管道，成本不合理。
> 修改目标：默认 Mode A（提取 + 独立覆盖审计），长文自动升级，人审发现遗漏时升级。覆盖报告为独立审计而非自检。降低平均 LLM 调用次数的同时保证覆盖报告的可靠性。解决策展悖论——人在不熟悉领域中无法判断提取遗漏的结构性困境。

---

## 一、核心原则

| 原则 | 含义 |
|------|------|
| **默认 Mode A（独立审计）** | 提取 + 独立覆盖审查 = 3 次 LLM/篇。覆盖审查与提取分离——飞行员和塔台是不同的人 |
| **可降级为 Mode A-fast** | 短源文（<3000 字）或用户对质量有信心时，可选择 2 次 LLM 快速路径（覆盖报告为自检） |
| **长文自动升级** | 字数 > 10000 是物理事实（可精确计算）——自动触发至少模式 B |
| **人审即质量门控** | 人审策展地图时发现质量问题 → 触发模式升级。这是语义质量的唯一判决点 |

---

## 二、决策逻辑

### 提取模式选择

```
if 源文字数 > 10000:
    → 至少模式 B
    原因：约 6000 tokens，一次 LLM 调用的穷举提取在此区间注意力衰减显著上升。这是可精确计算的物理事实，不是语义猜测。
else:
    → 模式 A（默认 3 次 LLM：提取 + 独立覆盖审查 + 策展）
    
    可选降级：
      if 源文字数 < 3000 且用户确认使用快速路径：
        → 模式 A-fast（2 次 LLM：提取含自检覆盖 + 策展）

人不做事前复杂度判断。Agent 不猜语义复杂度。

质量门控在人审：
  人审策展地图 + 覆盖报告
    → 质量通过 → SOP-002 策展入库
    → 发现遗漏 → 标记升级 → 模式 B/C 重新提取整篇
      （局部重提为 v2.3 能力）
```

### 为什么不做语义层面的复杂度判断
- **Agent 的判断不可靠**：复杂度是语义判断，Agent 在初步扫描时猜不准。高估 → 浪费 LLM 调用（不可逆）；低估 → 地图质量差 → 人审发现 → 重新提取（代价可逆）
- **默认 Mode A 没有「浪费」路径**：最差的后果是地图质量不够 → 重新提取。没有「高估复杂度浪费 LLM 调用」的路径
- **Mode A 的提取方式对大多数源文足够**：高质量模型在一次 LLM 调用中完成第 1-9 节的穷举提取表现已经很好

### 字数阈值统一说明

| 阈值 | 用途 | 触发行为 |
|------|------|---------|
| 3000 字 | Mode A-fast 可选上限 | ≤3000 可选择快速路径（覆盖报告为自检，2 次 LLM） |
| 5000 字 | 读取方式分界 | ≤5000 一次读完；>5000 分段注入 LLM 上下文 |
| 10000 字 | 提取模式触发 | 自动从 Mode A 升级到至少模式 B |
| 20000 字 | 物理限制（C6） | 读不完——多偏移采样 + 盲区声明（列出未细读的段落） |

三者关系在 SOP-001 Step 0.2 中统一说明。20000（C6）不被第 10 节取代——C6 声明「因读不完而未读」，第 10 节盲区声明「读了但判断不提取」。两者并存。

## 三、四种提取模式

### 模式对比

| | Mode A-fast | Mode A（默认） | Mode B（2 Pass） | Mode C（3 Pass） |
|---|---|---|---|---|
| **触发条件** | <3000 字 + 用户确认 | 默认；≤10000 字 | >10000 字 或 人审触发 | 人审发现 B 不够 |
| **适用源文比例（估计）** | 30-40% | 40-50% | 15-20% | ≤5% |
| **提取 LLM 调用** | 1 次 | 2 次 | 3 次 | 4 次 |
| **总 LLM 调用（含策展）** | 2 次 | 3 次 | 4 次 | 5 次 |
| **覆盖报告** | 自检（非独立） | **独立审计**（auditor 对照源文） | 交叉校验（assembler 对照 Pass 产出） | 交叉校验（assembler 对照 Pass 产出） |

### 各模式流程

```
Mode A（默认，3 次 LLM）：
  源文 → sop-001-modeA.md         → 第 1-9 节策展地图（1 次 LLM）
       → sop-001-modeA-auditor.md → 第 10 节覆盖报告（1 次 LLM，独立审查——对照源文检查覆盖。可用便宜模型）
       → 人审
         → 通过 → sop-002-curator.md → wiki 页面（1 次 LLM）
         → 不通过 → 升级到模式 B
  合计：3 次。覆盖报告为独立审计。

  覆盖审查的任务（比提取轻得多）：
    输入：源文 + 第 1-9 节提取结果
    任务：逐节对照覆盖度——不需要理解概念、不需要判断语义质量
    模型：可用便宜模型（Haiku / GPT-4o-mini）——这是覆盖匹配，不是语义理解

---

Mode A-fast（可选快速路径，2 次 LLM）：
  源文 → sop-001-modeA-fast.md     → 完整 10 节策展地图含自检覆盖报告（1 次 LLM）
       → 人审
         → 通过 → sop-002-curator.md → wiki 页面（1 次 LLM）
         → 不通过 → 升级到 Mode A（独立审计）或直接升级到 B
  合计：2 次。覆盖报告为自检。

  适用条件：
    - 源文 < 3000 字
    - 用户对单次提取质量有信心
    - 自检覆盖报告标注「非独立审计」
  
  实现：sop-001-modeA-fast.md 是独立文件——modeA.md 的提取逻辑 + 自检覆盖报告生成指令。
        不依赖 auditor。

---

模式 B（2 Pass，4 次 LLM）：
  源文 → sop-001-modeB-pass1-entities-claims.md → 实体+概念+论点（1 次 LLM）
       → sop-001-modeBC-pass2-relationships.md  → 关系清单（1 次 LLM）
       → sop-001-modeBC-assembler.md            → 轻量组装 + 覆盖报告（1 次 LLM）
       → 人审 → sop-002-curator.md              → wiki 页面（1 次 LLM）
  合计：4 次。assembler 对照两份独立 Pass 产出——覆盖报告为交叉校验。

---

模式 C（3 Pass，5 次 LLM）：
  源文 → sop-001-modeC-pass1-entities.md        → 实体+概念（1 次 LLM）
       → sop-001-modeBC-pass2-relationships.md  → 关系清单（1 次 LLM）
       → sop-001-modeC-pass3-claims.md          → 论点提取（1 次 LLM，有 Pass 1+2 上下文）
       → sop-001-modeBC-assembler.md            → 完整组装 + 覆盖报告（1 次 LLM）
       → 人审 → sop-002-curator.md              → wiki 页面（1 次 LLM）
  合计：5 次。assembler 对照三份独立 Pass 产出 + 重读源文。
```

### 模式 B 与模式 C 的本质区别

| | 模式 B（2 Pass） | 模式 C（3 Pass） |
|---|---|---|
| **提取策略** | Pass 1 合并「实体+概念+论点」——省一次 LLM。Pass 2 独立关系 | Pass 1「是什么」+ Pass 2「怎么关联」+ Pass 3「主张什么」——各司其职 |
| **优势** | 比 C 少 1 次 LLM；比 A 多独立关系轮次 | 准确性最高——Pass 3 在了解完整实体体系和关系网后提取论点 |
| **风险与缓解** | 合并提取可能将实体描述误判为论点。Prompt 内置自检：逐一核查每个论点是否为作者主张 | — |
| **assembler** | 轻量——不对照源文重读 | 完整——对照源文重读 |

### 成本估算

```
之前（一刀切 3 Pass + 组装）：
  每篇固定 5 次 LLM。20 篇 = 100 次

之后（默认 Mode A + 异常升级）：
  6 篇 Mode A-fast × 2 次  = 12 次（<3000 字短文，走快速路径）
  10 篇 Mode A × 3 次      = 30 次（默认路径）
  3 篇 Mode A 升级 B × 4 次 = 12 次（重新提取的额外成本）
  1 篇 Mode B × 4 次       = 4 次（>10000 字自动触发）
  合计：58 次

节省：约 42%。覆盖审计独立，可靠性不低于之前。
```

## 四、策展地图新增第 10 节：覆盖报告

### 为什么需要覆盖报告

策展悖论：人在不熟悉领域中无法判断策展地图是否有遗漏。覆盖报告提供**不需要领域知识的覆盖异常信号**。

### 覆盖报告的独立性

| 模式 | 覆盖报告性质 | 生成者 | 可信度 |
|------|------------|--------|--------|
| **Mode A（默认）** | **独立审计** | `sop-001-modeA-auditor.md`——独立审查 Agent。输入源文 + 提取结果，只做覆盖匹配，不做语义判断 | 高——审查 Agent 和提取 Agent 是两次独立 LLM 调用，不共享上下文 |
| **Mode A-fast** | **自检** | `sop-001-modeA.md` 内部生成。提取和覆盖报告同一调用 | 低于独立审计。LLM 在某节漏提时也可能漏报异常。标记了异常更可能是真的 |
| **Mode B/C** | **交叉校验** | `sop-001-modeBC-assembler.md` 对照独立 Pass 产出生成 | 高——assembler 不参与提取 |

### 独立审计的工作方式（Mode A 默认）

```
sop-001-modeA-auditor.md 的任务（轻量级审查）：

  角色：你是策展地图的覆盖审计员。你不提取新内容，不了解领域知识。
        你的唯一任务是逐节检查——提取结果是否覆盖了源文的每个段落。

  输入：
    - 源文全文
    - 第 1-9 节策展地图（提取结果）

  任务（纯覆盖匹配，不需要语义理解）：
    1. 源文有多少个大节/自然段落？
    2. 每个节中，提取项覆盖了多少名词/概念/实体提及？
    3. 有没有提取数为 0 的描述性段落（不是代码/引用/附录）？
    4. 有没有全景概括中提到、但提取项中找不到对应条目的概念？
    5. 有没有密度显著偏低的节？

  不要做的事：
    ✗ 不判断提取的概念定义是否正确
    ✗ 不判断关系提取是否准确
    ✗ 不提取任何新内容
    ✗ 不评估语义质量

  输出：10.x 格式的覆盖报告（见 extraction-interface.md 第 10 节）

  推荐模型：便宜模型（Haiku / GPT-4o-mini / DeepSeek-V3-Lite）。
           这是覆盖匹配任务，不需要深度推理。
```

### 格式规范（extraction-interface.md 第 10 节）

```
10. 覆盖报告

10.1 元信息

  生成模式：[独立审计 — Mode A] | [自检 — Mode A-fast] | [交叉校验 — Mode B/C]
  审查 Agent：[提取 Agent 同一调用] | [独立 auditor] | [assembler]
  可信度说明：
    独立审计 — 审查 Agent 与提取 Agent 分离，对照源文独立检查覆盖。
    自检      — 提取与覆盖报告同一次 LLM 产出，异常标记较保守。
    交叉校验  — assembler 对照独立 Pass 的提取输出，非同一认知过程。

10.2 节级覆盖

  | 源文节 | 原文首句/标题 | 全景概括提及 | 提取项数 | 覆盖状态 | 说明 |
  |--------|-------------|------------|---------|---------|------|
  | ... | ... | ... | ... | ... | ... |

  覆盖统计：源文共 N 个大节，提取覆盖 M 个，K 个标记为盲区。总提取项数：X

10.3 全景概括对照（Mode A / Mode A-fast 填写——两者均有全景概括可对照。Mode B/C 的 assembler 不从全景概括出发，可省略此节）

  全景概括逐段描述 vs 实际提取覆盖的差异：
    - 第 N 节概括提到「X、Y、Z」→ 提取项缺「Z」→ ⚠ 交叉校验异常

10.4 密度异常

  均值 = 总提取项数 / 覆盖的节数（排除盲区）
  某节 > 3×均值 → "密集"。某节 < 0.3×均值 且非盲区 → "偏低"

10.5 结构一致性

  源文大节数 vs 覆盖大节数。覆盖率 < 80% → ⚠

10.6 盲区声明

  | 位置 | 原因 | 内容类型 | C6 相关？ |
  |------|------|---------|----------|
  | ... | ... | ... | ... |

  注：C6 超长盲区（读不完）与本表盲区（读了但判断不提取）独立列出。
```

## 五、修改后的人审流程

```
策展地图头部提示：
  > ⚠ 请先阅读第 10 节「覆盖报告」（结构覆盖概览，约 30 秒），再逐节审核内容。
  > 覆盖报告为 [独立审计 — Mode A] / [自检 — Mode A-fast] / [交叉校验 — Mode B/C]。

---

Step 1：看覆盖报告（30 秒）——不需要领域知识

  ─ 看「10.1 元信息」
    · 独立审计 → 可信度高
    · 自检 → 异常信号较保守，如有异常建议升级重审
    · 交叉校验 → 可信度高

  ─ 看「10.2 节级覆盖」中有没有提取数为 0 的节？
    → 是代码/引用/附录 → 通过。是文字描述段 → ⚠ 异常

  ─ Mode A / A-fast 用户看「10.3 全景概括对照」
    → 概括提到但提取缺的 → ⚠

  ─ 看「10.4 密度异常」有没有偏低且非盲区的节？

  ─ 看「10.5 结构一致性」覆盖率 < 80% → ⚠

Step 2：看策展地图内容（正常审核）

  ─ 概念分类、关系提取、聚类合理？→ 确认/调整
  ─ 标记「确认入库」/「忽略」/「需要更多源」

Step 3：决定是否升级

  Mode A-fast（自检覆盖）下，以下任一 → 升级到 Mode A（独立审计）或直接 B：
    ─ 覆盖报告标记了 ⚠
    ─ 全景概括对照发现交叉校验异常

  Mode A（独立审计）下，以下任一 → 升级到模式 B：
    ─ 覆盖报告发现 ≥2 个非代码/引用的盲区
    ─ 你在 Step 2 发现关系提取明显不充分

  Mode B 下 → 升级到模式 C：
    ─ 发现论点提取存在误判（实体描述被误认为论点）

  升级 = 整篇源文用更高模式重新提取。局部重提为 v2.3 能力。
```

## 八、异常处理

```
长度 > 10000 字时 Agent 不确定 B 还是 C：
  → 用 B（保守）。人审发现论点误判 → 升级 C

Mode A auditor 调用失败：
  → 降级为自检覆盖（Mode A-fast 路径）
  → Agent 签名段注明降级原因

Mode B/C 某个 Pass 失败：
  → 盲重试 2 次
  → 仍失败 → 已完成的 Pass 保留，缺失的标注在覆盖报告盲区中
  → 报告用户决定

覆盖报告生成格式异常：
  → 不阻塞策展地图交付
  → Agent 签名段注明异常 → 降级为人工逐段对照

Mode A-fast 自检覆盖发现异常：
  → 默认升级到 Mode A（先试独立审计）或直接 B
  → 自检异常比自检正常更可信（LLM 倾向不标记自己盲区）

Mode B Pass 1 论点自检发现 ≥3 个存疑项：
  → 建议升级到模式 C
```

## 九、与策展悖论的关系

```
策展悖论：人在不熟悉领域中无法判断提取是否有遗漏。

Mode A（默认）的解法：
  提取和覆盖审查分离——auditor 是独立 Agent，不共享提取 Agent 的上下文。
  auditor 不需要理解领域知识——它只做覆盖匹配。
  就像飞行员做飞行计划（提取），塔台做雷达扫描（覆盖审查）——不同的人，不同的仪表。

Mode A-fast 的局限：
  提取和覆盖报告同一调用——自检。适用于短文且用户对质量有信心。
  在文档中明确标注非独立审计——用户知情选择。
```

## 十一、验证标准

1. **Mode A 独立审计路径可独立运行**——提取 → auditor → 策展，三个调用依次完成
2. **auditor 可用便宜模型**——覆盖匹配任务不要求深度推理，Haiku/GPT-4o-mini 级模型即可
3. **四种模式产出格式一致**——都输出 10 节策展地图，SOP-002 不感知模式差异
4. **覆盖报告不依赖领域知识**——异常信号是结构性的
5. **人审负担下降**——先看覆盖报告再看内容
6. **升级路径清晰**——A-fast 自检异常 → A 独立审计 → B → C
7. **降级路径完整**——auditor 失败降级 A-fast → 覆盖报告失败降级人工对照
8. **Mode A 实测**——用 `examples/` 样例实测，如果遗漏率显著高于 B/C，调整阈值
9. **文档一致**——sop-v2-full.md / build-plan.md / vision.md / README 一致
