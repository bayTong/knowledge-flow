## 一、核心理念与原则

### 什么是外置第二大脑

不是「AI 帮你记笔记」，而是 **AI 智能体作为持续运行的认知外挂**：

- **摄入管线**：AI 穷举提取 → 人审 → 策展入库。人做价值判断，AI 做不遗漏的机械提取
- **持久记忆**：知识库作为 AI 的跨会话长期记忆。AI 在对话中主动检索已有知识，而非从零开始
- **元认知监控**：独立 Agent 定期扫描知识库健康度——遗漏、漂移、盲区、结构趋势
- **自我进化**：系统从人的审核反馈中学习自己的盲区，持续改进提取和策展质量

### 核心原则

| 原则 | 含义 |
|------|------|
| **AI 做覆盖，人做判断** | AI 穷举提取不遗漏；人决定什么重要、什么该入库 |
| **写入和检查分离** | 摄入 Agent 做提取+写入（摄入会话）；独立 Cron Agent 做语义检查和趋势分析（独立会话）。避免上下文污染 |
| **成本分层** | 最终产出用最好的模型；辅助检测用便宜模型；机械任务不用 LLM |
| **只报告，不自动修** | MC-001 发现的问题只标记和报告，不自动修改内容。所有「该不该做」交给人决策 |
| **SCHEMA 事后涌现** | SCHEMA 不前置定义——只从通用模板起步，随每次摄入从原料中自动建议新标签、新模块，经人确认后注册 |

## 四、LLM 的四项根本局限与系统对策

这四个局限是当前 LLM 技术的硬边界，架构不能「解决」它们，只能**降低伤害**、**把不可消除的部分交还给人类**

### 局限 1：无真正语义理解

> LLM 能关联文字，但不能「理解」概念间的深层逻辑关系。做的是统计模式匹配，不是推理。

**架构对策**：

| 对策 | 阶段 | 机制 |
|------|------|------|
| 多轮次提取（分治降低认知负担） | 1 | Mode A 默认单次提取 + 独立覆盖审计；>10000 字自动升级 Mode B；人审触发 Mode C |
| 提取接口（强迫系统化扫描） | 0 | 结构化输出格式消除「跳读→概括」的统计噪音 |
| 聚类辅助（发现 LLM 未标注的关联） | 1 | Embedding 聚类补充 LLM 的关联遗漏 |
| MC-001 独立 Agent 交叉验证 | 2 | 维度 2：独立 Agent 评估提取质量 |

### 局限 2：长期连贯思考丢失

> 跨数周的复杂推理链会丢失上下文。压缩、摘要、向量检索都会丢细节。

**架构对策**：

| 对策 | 阶段 | 机制 |
|------|------|------|
| 知识库作为外部持久记忆 | 已有 | wiki 页面 + log.md + index.md |
| MC-001 趋势视图 | 2 | 维度 4+5：跨越时间的统计和趋势 |
| 概念漂移追踪 | 2 | log.md 标记 → MC-001 聚合报告 |
| 主动检索（注入上下文） | 3 | Agent 识别概念 → 自动检索 KB → 融入回复 |

### 局限 3：无自主价值判断

> LLM 不能判断「这个方向值不值得深挖」

**架构对策**：

| 对策 | 阶段 | 机制 |
|------|------|------|
| 人审设在关键决策点 | 已有 | SOP-001 策展地图审核 |
| MC-001 只报告不决策 | 2 | 所有「该不该」交给人 |
| 聚类建议只做提示 | 1 | 「建议合并」不是「应该合并」 |

### 局限 4：无真正创造性连接

> LLM 只在表层做相似度匹配，不能跨领域产生新洞见

**架构对策**：

| 对策 | 阶段 | 机制 |
|------|------|------|
| MC-001 跨页面/跨 KB 关联发现 | 2 | AI 做大规模比对（不遗漏） |
| 人做创造性判断 | 2 | 人从 AI 的关联建议中选出真正有意义的 |
| 联合索引降低发现门槛 | 3 | 跨 KB 的统一概念索引 |

- 每个 prompt 模板自包含，标注输入、输出格式、硬约束

- `extraction-interface.md` 是权威的格式规范，所有 prompt 模板以它为准

### 5.2 `scripts/` 目录 — 参考实现脚本（P1 强烈建议）

纯 Python 标准库实现，零外部依赖（`pathlib`, `re`, `json`, `datetime`）。

| 文件 | 功能 | 说明 |
|------|------|------|
| `scripts/lint.py` | 知识库 Lint 扫描器 | 扫描 wiki/ 目录。检查：frontmatter 完整性、标签合规（对照 SCHEMA 注册表）、wikilink 格式（pipe 格式 `[[x\|y]]`）、断链、孤页、超标页。输出：分类报告（Error / Warning / Notice）。支持 `--json` |

**设计要点**：
- 每个脚本独立可运行：`python scripts/lint.py /path/to/kb`
- 支持 `--json` 机器可读输出（供 MC-001 Cron 消费）
- 参考实现——用户可以根据自己的 LLM 工具适配

## 六、阶段 1：自适应提取分层 + 覆盖报告 ✅ v2.2

**目标**：默认单次提取 + 独立覆盖审计。长文自动升级。人审发现遗漏时升级。降低平均 LLM 调用次数，同时通过覆盖报告解决策展悖论。

详细修改方案见 [`docs/adaptive-extraction-plan.md`](adaptive-extraction-plan.md)。本阶段已在 v2.2 实施，以下为摘要。

### 6.1 四种提取模式

| 模式 | LLM 调用（含策展） | 覆盖报告 | 触发条件 |
|------|-------------------|---------|---------|
| **Mode A（默认）** | 3 次 | 独立审计 | 默认路径；≤10000 字 |
| **Mode A-fast** | 2 次 | 自检 | < 3000 字 + 用户确认 |
| **Mode B（2 Pass）** | 4 次 | 交叉校验 | > 10000 字 或 人审触发 |
| **Mode C（3 Pass）** | 5 次 | 交叉校验 | 人审发现 B 不够 |

### 6.2 策展地图新增第 10 节「覆盖报告」

覆盖报告提供不需要领域知识的覆盖异常信号（节级覆盖、密度异常、盲区声明、全景概括对照）。Mode A 中由独立 auditor 生成（提取与覆盖审查分离），不新增 LLM 调用——auditor 推荐便宜模型。

人审时**先看覆盖报告（30 秒），再看内容**。覆盖异常信号是结构性的——人不需要理解领域就能发现「第 4 节 0 条提取」。

### 6.3 聚类辅助审核

**推迟到后续版本。** 覆盖报告作为人审第一道防线（即刻可用，不需要 embedding 基础设施）。聚类辅助审核作为 v2.3 或 MC-001 维度之一（跨页面语义关联发现已部分覆盖此功能）。

### 6.4 模型选择指导

| 步骤 | 推荐模型级别 | 理由 |
|------|------------|------|
| Mode A 提取（sop-001-modeA.md） | 中-高 | 单次调用完成所有提取 |
| Mode A auditor（sop-001-modeA-auditor.md） | 低（Haiku 级别） | 覆盖匹配不要求深度推理 |
| Mode B/C Pass 1 实体提取 | 中-高 | 实体识别需要广域知识 |

## 七、阶段 2：MC-001 元认知 Agent

**目标**：建立独立于摄入管道的知识库健康监控系统。定时产出趋势报告和仪表盘。

### 7.1 定位

- **执行者**：独立的 Cron job（独立会话，不共享摄入 Agent 的上下文）
- **频率**：每周日凌晨 3:00（或按需调整）
- **权限**：只标记和报告，不自动修改 wiki 页面内容
- **预算**：每周期 LLM 调用有上限（质量抽检 ≤5 次，遗漏审查 ≤3 次）
- **实现**：Hermes Cron job + `kb-metacognition` skill + 参考脚本

### 7.2 六个维度

#### 维度 1：遗漏检测

```
输入：最近 N 篇入库的源文 + 对应的策展地图 + wiki 页面
流程：
  1. 源文段落切分 → embedding
  2. 策展地图提取项 → embedding
  3. 余弦相似度比对 → 标记「段落级遗漏候选」
  4. 对候选段落，独立 LLM 做句级遗漏审查
     （独立 Agent 避免了摄入 Agent 的上下文污染）
输出：「以下段落/句子可能未被充分提取 → 建议复核」
成本：embedding 零 LLM 成本 + 每周期最多 3 次 LLM 调用

局限：Embedding 只能检测段落级遗漏，不能检测句级遗漏。
      句级遗漏需要 LLM 二次审查或人审来补。
```

#### 维度 2：语义质量抽检

```
输入：随机抽取 10% wiki 页面
流程：
  独立 Agent 评估每个页面：
    - 这个页面完整吗？有没有明显遗漏的概念？
    - 结构合理吗？是否存在该拆分但没拆分的？
    - wikilink 连接是否充分？
输出：质量报告 + 改进建议（按严重度排序）
预算：每周期最多 5 次 LLM 调用
模型：可用便宜模型（如 Haiku / GPT-4o-mini），抽检是辅助性的
```

#### 维度 3：跨页面语义关联发现

```
输入：所有 wiki 页面的核心概念（从 frontmatter + 首段提取）
流程：
  1. 对所有页面对计算 embedding 相似度
  2. 标记：高相似度（>0.85）但无 wikilink 连接的页面对
  3. 排除已知关系（已有直接 wikilink 的）
输出：「这些页面讨论高度相似的概念但未被连接 → 建议创建 wikilink」
成本：纯 embedding 计算，零 LLM 成本
      N=50 页面 → O(n²) = 1225 次比较，几秒完成
```

#### 维度 4：概念漂移聚合

```
输入：log.md 中所有 [概念漂移] 标记
流程：
  1. 扫描 log.md 解析漂移标记
  2. 统计：多少漂移被标记？多少仍 unresolved？
  3. 按模块/概念分组
输出：漂移趋势报告
      「过去一个月 3 个概念漂移，2 个仍未解决」
成本：零 LLM（纯 log.md 文本扫描）
```

#### 维度 5：结构健康趋势

```
输入：wiki/ 目录 + index.md + 历史统计快照
流程：
  1. 统计：页面数、孤页数、超标页数、断链数
  2. 对比上一周期的数据 → 趋势线
  3. 按模块分解
输出：趋势报告
      「孤页从 5 减少到 3（↓40%）；超标页从 7 增加到 9（↑29%）」
成本：零 LLM（纯规则统计）
```

#### 维度 6：更新知识库健康仪表盘

```
技术方案：同 hermes-evolution.html 的数据驱动 HTML 模式
  - <script id="health-data" type="application/json"> 持有数据
  - JavaScript 构建 DOM
  - 零外部依赖（系统字体，file:// 可用）
数据源：MC-001 维度 1-5 的结构化输出
```

## 十、设计决策记录

### 为什么 MC-001 是独立 Cron 而非嵌入 SOP

- **上下文污染**：摄入 Agent 的注意力分布被提取历史污染，语义检查（遗漏、漂移、质量）的判断不可靠
- **时间视角不同**：SOP-003 是快照检查（「现在有没有问题」），MC-001 是趋势分析（「在变好还是变差」）
- **非阻塞**：深度语义检查不应阻塞日常摄入管道
- **GBrain 验证**：Dream Cycle 独立于写入 Agent 的设计在 146K 页面规模上已被验证

### 为什么三 Pass 分治优于一次性提取

- **认知负荷管理**：一次性要求 LLM 同时识别实体、判断关系、提取论点 → 注意力分散，三个维度互相稀释
- **三轮分治**：每轮只关注一个维度 → 每轮的提取深度和完整度都更高
- **组装阶段独立**：缺口分析、SCHEMA 建议需要看到全部提取结果才能判断——任何单个 Pass 没有这个全局视野。组装阶段作为独立的第 4 次 LLM 调用，拥有三份 Pass 的完整输出
- **可审计**：三轮产出独立可检查 → 如果关系提取质量差，可以只调整 Pass 2 的 prompt，不影响 Pass 1 和 3

### 为什么聚类不能替代提取但能辅助审核

- **聚类不是提取**：Embedding 只能判断「相似」，不能填写 7 字段（定义、类型、不确定性、关系）
- **聚类辅助审核**：在提取完成后，聚类帮助人发现 LLM 未显式标注的隐含关联，减少人审时的认知负担

### 为什么 SCHEMA 是事后涌现而非前置定义

- SOP-000 只用通用模板（目录结构 + 命名规则 + 通用标签），不包含领域特定内容
- 每次 SOP-001 策展地图扫描到新概念 → 建议 SCHEMA 新增标签/模块
- SOP-002 入库时注册新标签 → SCHEMA 随每次摄入「长出来」
- 这确保 SCHEMA 反映的是实际存在的知识结构，而非人为预设的分类

### 为什么策展地图比粗读器更适合作为名称

- 「粗读器」描述的是机制（谁来读），关注输入侧
- 「策展地图」描述的是产出（地图），关注输出侧——用户直接使用的东西
- 「策展」与 SOP-002「策展入库」术语一致，形成清晰的 pipeline 语义
