# 策展悖论

> KnowledgeFlow 的核心差异化论述。所有设计决策——两阶段管道、7 字段接口、硬约束 C1-C7——都可追溯到这一悖论。

---

## 悖论

两个各自合理的前提，放在一起构成了一个悖论：

**前提 1**：高质量的知识策展需要领域判断力。你需要区分核心概念和次要细节，识别两个不同表述是否指向同一事物，判断哪些实体值得作为领域核心提取、哪些只是边缘提及，以及知识之间应该如何关联。这些判断只有真正理解领域的人才能做出。

**前提 2**：人们采用知识管理工具，正是因为还不懂这个领域。目标是更快、更直观、更轻松地从陌生材料中学习——构建结构化知识或实现一系列交互能力。这个过程高度依赖 AI 进行分析和推理。

悖论本身：**策展的责任在人（只有你知道自己的目标和上下文），但人缺乏策展所需的领域知识；而有知识处理能力的实体（LLM）却缺乏判断「什么对你重要」的上下文，且其风险——提取遗漏、幻觉——无法忽视。**

两个独立有效的前提指向了一个矛盾：到底该由谁来策展？

---

## 两种常见的失败模式

当 LLM 被放在策展者的位置上时，两种失败模式占主导：

### 遗漏

LLM 判断某个概念不够重要，跳过它。没有原文作为 ground truth，你永远不知道被漏掉了什么。这是不可修复的失败——噪音可以几秒内删除，但遗漏是不可见的。

### 过度简化

LLM 产出了一篇读起来不错的 wiki 页面，但你无法判断它代表了原文的全部内容，还是只是 LLM 选择包含的子集。你失去了掌控感——输出看起来合理，但你无从衡量原文和产物之间的差距。

核心洞察：**问题不在于 LLM 的能力不足——而在于角色的错配。** LLM 的优势（穷举扫描——不遗漏任何东西）在它被迫同时判断重要性时被边缘化了。它的弱点（判断什么对你重要）被推到了最前线。设计目标不是「更好的策展算法」——而是**把判断权拉回人类一侧，将 LLM 从策展者降级为提取工具，并证明在这个角色中，LLM 可以比人类更可靠、更彻底地完成任务。**

---

## 常见工具的解法（以及为什么它不够）

大多数 AI 知识管理工具通过忽略前提 2 来解决这个张力——它们直接让 LLM 策展。LLM 阅读原文，决定什么是「核心概念」、什么是「次要细节」，创建页面，分配标签，撰写摘要。

这很快、摩擦很低，但有一个无法修复的缺陷：**LLM 的遗漏远比 LLM 的噪音更难检测。** 如果 LLM 过度提取（噪音），你可以在几秒内删除不需要的页面。如果 LLM 悄悄跳过了关键概念（遗漏），你永远不知道它被漏掉了——因为没有策展地图存在，原文和成品页面之间的推理过程完全在 LLM 的黑箱里。

---

## KnowledgeFlow 的答案

KnowledgeFlow 不试图让 LLM 更擅长判断。它重构了责任分配：

| 角色 | 职责 | 约束 |
|------|------|------|
| **LLM（策展地图生成）** | 穷举机械提取——每个概念、实体、关系、事实主张，锚定到原文精确位置 | C5：绝不过滤重要性。提取一切 |
| **人（审核）** | 在策展地图检查点做语义策展——标记「入库」「忽略」或「需要更多源」 | 策展地图是强制审计面 |
| **LLM（策展入库）** | 受约束写入——只处理人确认的条目，在 SCHEMA 约束下执行 | C1：粗读阶段绝不创建 wiki 页面 |

两阶段管道不是为了效率——是为了**可审计性**。你不需要信任 LLM 的判断。你只需要信任它提取了一切（可通过逐节覆盖和原文引用验证），然后将自己的判断应用于其余部分。

**v2.2 新增**：覆盖报告（策展地图第 10 节）提供了不需要领域知识的覆盖异常信号——节级覆盖、密度异常、全景概括对照。人在不熟悉领域中无法判断遗漏，但可以看到「第 4 节 0 条提取」并定位到原文核查。覆盖报告是仪表盘——你不需要懂空气动力学，看到高度计骤降就知道有问题。

## 设计推导

每一个硬约束都追溯到这一悖论：

| 约束 | 防止什么 |
|------|---------|
| **C1：粗读阶段不创建 wiki 页面** | LLM 绕过人审直接写入 |
| **C2：每条提取必须有原文引用** | 不可验证的黑箱提取 |
| **C3：不确定性必须显式标记** | LLM 对不理解的东西假装确定 |
| **C5：绝不过滤重要性** | 遗漏问题——LLM 悄悄跳过它认为「不重要」的内容 |
| **C6：超长原文必须列出未读段落** | 覆盖幻觉——你以为地图完整，实则不然 |

7 字段提取接口之所以存在，是因为没有结构化字段的提取是无法验证的散文。5 种不确定原因分类之所以存在，是因为不同的不确定来源需要不同的人类处理策略。三层防御之所以存在，是因为格式漂移、结构衰减和 SCHEMA 变更的涟漪效应是三种不同的失败模式，没有单一检查能够全部捕获。

## 为什么这对作品展示很重要

当把这个项目展示给技术受众（招聘经理、同行工程师）时，策展悖论是它区别于「又一个笔记工作流」的东西。大多数项目描述的是它们*做了什么*。这个项目解释的是*为什么架构长这样*——每个设计选择都追溯到一个清晰陈述的问题。悖论本身是差异化因素：

> 「我识别出了 LLM 辅助知识管理中的根本张力，并设计了一个架构来解决它——不是通过让 LLM 更聪明，而是通过重构人-LLM 的责任边界。」
