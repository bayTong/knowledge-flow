## 关系类型

| 关系类型 | 含义 | 示例 |
|---------|------|------|
| `uses` | A 使用 B 作为组件/工具 | Agent 使用 RAG 作为其核心工具之一 |
| `complements` | A 和 B 互补，各自解决不同维度 | RAG（知识）和 Agent（行为）互补 |
| `contains` | A 包含 B 作为子集/子组件 | 多智能体系统包含多个单智能体 |
| `contrasts_with` | A 和 B 在某个维度上有对比/差异 | BM25 检索 vs 知识图谱 RAG |
| `enables` | A 赋能 B——为 B 提供底层能力或基础 | 知识图谱为 Agent 规划提供推理基础 |
| `improves` | A 提升 B 的效果/性能 | 重排序提升初步检索的 MRR 15-40% |
| `evolves_from` | B 是 A 的技术演进/升级 | CoT → ToT 的范式演进 |
| `serves_as` | A 充当 B 的角色/功能 | 知识图谱充当 Agent 长期记忆存储 |
| `optimizes` | A 优化 B 的执行路径 | 意图识别层优化 Function Calling 调用路径 |

如果以上类型无法准确描述某条关系，自行扩展新类型并显式标注「新增关系类型」。

## 提取规则

### 显式关系
- 原文明确说 A 和 B 有关系——必须提取
- 置信度设为「确定」
- 标注原文依据（精确到段落/章节）

### 隐式关系
- 你通过推理判断 A 和 B 可能存在关系——可以提取，但必须：
  - 标注「推测」置信度（不得设为「确定」）
  - 给出推测依据：「因为 A 和 B 在原文中并列讨论……」
  - 放在独立的「隐式关系」小节中

> 主体 A 和 B 必须是 Pass 1 中已提取的实体——使用 Pass 1 中的标识符引用。如果发现原文存在 Pass 1 遗漏的实体，可补充提取并标注「Pass 1 遗漏补充」，但这不是本轮主要任务。

## 重要提醒

1. **每条关系必须能在原文中找到依据**——显式关系引用原文句子，隐式关系给出推理链
2. **不确定就放隐式**——宁可按推测提交，也不将隐式关系伪装成显式
3. **Pass 2 的输出将作为后续阶段的输入**——关系信息帮助 assembler 在完整上下文中生成覆盖报告和分析
4. **输出后不做任何其他操作**。你的输出将作为 `{{PASS2_OUTPUT}}` 传递给 assembler（`sop-001-modeBC-assembler.md`）
