# 提取接口技术规范

> 本文档定义了 SOP-001 策展地图中所有提取项和覆盖报告的数据格式、字段约束、边界情况处理。所有 prompt 模板（sop-001-modeA.md、sop-001-modeA-fast.md、sop-001-modeA-auditor.md、sop-001-modeB-pass1-entities-claims.md、sop-001-modeBC-pass2-relationships.md、sop-001-modeC-pass1-entities.md、sop-001-modeC-pass3-claims.md、sop-001-modeBC-assembler.md）的提取和覆盖报告格式以本文档为权威参考

---

## 一、实体/概念提取接口（7 字段）

每条实体/概念提取必须包含全部 7 个字段：

| # | 字段 | 类型 | 说明 | 约束 |
|---|------|------|------|------|
| 1 | `标识符` | string | 唯一 ID，格式 `@类型:简称` | 全小写英文，连字符分隔。类型前缀：`concept`（核心概念）、`tool`（工具/框架）、`model`（模型）、`platform`（平台/产品）、`storage`（存储/索引）。扩展新类型时显式标注 |
| 2 | `名称` | string | 人类可读名称，可附带括号说明 | 优先使用原料中的原词；括号补充用于消歧（如 `RAG（检索增强生成）`） |
| 3 | `所属层次` | enum | 从步骤 2 确定的层次中选择 | 必须在已声明的层次列表中 |
| 4 | `一句话描述` | string | 原料中对该实体的定义或核心描述 | 优先从原文直接引用或提炼；不添加原文没有的信息 |
| 5 | `原文引用` | string | 精确到段落/章节/行号 | 不允许模糊引用（「全文多处」）；格式：`第 X 节`、`第 X 段`、或 `行号 Y-Z` |
| 6 | `置信度` | enum | `确定` / `推测` / `不确定` | — |
| 7 | `不确定原因` | enum/null | 仅当置信度 ≠「确定」时填写 | 从 5 种分类中选择一个；值为 `—` 表示确定项 |

### 字段约束详解

**标识符**：
- 格式：`@类型:简称`，如 `@concept:rag`、`@tool:langchain`
- 简称使用英文小写连字符，取最具代表性的关键词
- 同一地图内标识符必须唯一
- 多轮次提取时，Pass 1/2/3 使用一致的标识符引用同一实体

**名称**：
- 中文实体直接用中文名，英文实体保留英文名
- 括号补充不超过 15 个字符
- 示例：`RAG（检索增强生成）`、`Pinecone`、`DeepSeek V4 Pro`

**原文引用**：
- 必须足以让读者在原料中定位到原句
- 超长原料（> 20000 字）可用行号范围
- 不写「全文多处」「多处提到」——每处引用是独立的提取项

## 二、不确定原因（5 种分类）

| 类别 | 含义 | 典型场景 | 人类处理策略 |
|------|------|---------|------------|
| `来源不可靠` | 原文中的数值/声明/引用缺乏原始出处 | 「MRR 提升 15-40%」但未引用具体论文 | 查找原始论文验证 |
| `Agent 理解局限` | 提取者不确定自己的理解是否正确 | 语义模糊的段落、高度技术性的内容 | 人审时重点关注，必要时查阅其他来源 |
| `原文模糊` | 原料本身的表述不够明确 | 类比过于简化、用日常语言描述技术概念 | 接受模糊性，入库时标注 |
| `信息不完整` | 原文提到了但未展开 | 「此外还有 Graph RAG……」一句话带过 | 标记为「待更多原料」，后续找专门资料 |
| `总结压缩损失` | 原料本身是二手总结，可能丢失了原始对话/文档的细节 | 对话总结而非原始记录 | 尽量找一手资料；没有则接受 |

### 分类选择指南

```
数值/数据声明没有引用 → 来源不可靠
你读了几遍还没完全理解 → Agent 理解局限
写得含糊、有很多「可能」「大概」 → 原文模糊
提到了一嘴就跑了 → 信息不完整
这本身就是个摘要/总结 → 总结压缩损失
```

## 三、关系提取接口

| # | 字段 | 类型 | 说明 |
|---|------|------|------|
| 1 | `标识符` | string | 格式 `@rel:描述`，如 `@rel:agent-use-rag` |
| 2 | `主体 A` | string | 关系的主体，引用已提取实体的标识符或名称 |
| 3 | `关系类型` | enum | 从预定义类型中选择 |
| 4 | `主体 B` | string | 关系的客体，引用已提取实体的标识符或名称 |
| 5 | `类型` | enum | `显式` / `隐式` |
| 6 | `原文依据` | string | 显式：引用原文句子或段落；隐式：写推测推理链 |
| 7 | `置信度` | enum | `确定` / `推测`。隐式关系不得为「确定」 |

### 关系类型定义

| 类型 | 方向 | 含义 | 示例 |
|------|------|------|------|
| `uses` | A → B | A 使用/调用 B 作为组件或工具 | Agent 使用 RAG 作为核心工具 |
| `complements` | A ↔ B | A 和 B 互补，各自解决不同维度 | RAG（知识）与 Agent（行为）互补 |
| `contains` | A → B | A 包含 B 作为子集/子组件 | 多智能体系统包含多个单智能体 |
| `contrasts_with` | A ↔ B | A 和 B 在某个维度上有对比/差异 | BM25 vs 知识图谱 RAG |
| `enables` | A → B | A 为 B 提供底层能力或基础 | 知识图谱为 Agent 规划推理提供基础 |
| `improves` | A → B | A 提升 B 的效果/性能（通常有量化数据） | 重排序提升初步检索的 MRR 15-40% |
| `evolves_from` | B → A | B 是 A 的技术演进/升级 | CoT → ToT 范式演进 |
| `serves_as` | A → B | A 充当 B 的角色/功能 | 知识图谱充当 Agent 长期记忆存储 |
| `optimizes` | A → B | A 优化 B 的执行路径（非性能提升，而是路径优化） | 意图识别层优化 Function Calling 调用路径 |

### 隐式关系的提取规则

- 必须标记为「隐式」
- 必须标注推测依据（一句话说明推断逻辑）
- 置信度不得为「确定」——只能写「推测」
- 与其他显式关系隔离输出（独立小节）

## 四、论点/主张提取接口

| # | 字段 | 类型 | 说明 |
|---|------|------|------|
| 1 | `标识符` | string | 格式 `@thesis:xxx` / `@claim:xxx` / `@rec:xxx` |
| 2 | `主张类型` | enum | `论点` / `事实主张` / `方法论建议` |
| 3 | `主张内容` | string | 精确复述或转述作者的原文 |
| 4 | `原文引用` | string | 精确到段落/章节 |
| 5 | `置信度` | enum | `确定` / `推测` / `不确定` |
| 6 | `不确定原因` | enum/null | 同上 5 种分类 |
| 7 | `可验证性` | enum | `可验证`（有明确数据/出处）/ `可查证`（可找到原始出处）/ `不可验证`（主观观点） |

## 五、置信度定义

| 级别 | 含义 | 判定标准 |
|------|------|---------|
| `确定` | 提取的内容与原料一致，不存在歧义 | 原文表述清晰 + 你的理解有把握 + 来源可靠 |
| `推测` | 提取的内容可能是对的，但存在一定不确定性 | 原文表述不够清晰 / 你做了推理 / 来源是二手的 |
| `不确定` | 提取的内容存在显著不确定性 | 原文模糊 / 信息不完整 / 来源明显不可靠 |

> 置信度衡量**提取准确性**（你是否正确理解了原料说了什么），而非**来源可靠性**（原料说的是不是真的）。两者正交——一份博客说「GPT-5 参数量是 GPT-4 的 10 倍」可以被「确定」地提取（你正确理解了这句声明），同时标注不确定原因为「来源不可靠」（这句话本身可能不可信）

## 六、策展地图产出约束

**必须有的 10 节**：全景概括 → 提取层次说明 → 实体清单 → 关系清单 → 事实主张 → 不确定标记 → 缺口分析 → SCHEMA 建议 → Agent 签名 → 覆盖报告

**Agent 签名必须包含**：提取统计（各类条目的数量）+ 提取模式（Mode A/A-fast/B/C）+ 覆盖报告生成方式（独立审计/自检/交叉校验）+ 未创建 wiki 页面声明 + 等待审核标记

**全局约束**：
- ☒ 不创建任何文件（除策展地图本身）
- ☒ 不更新 index.md
- ☒ 不修改已有页面
- ☑ 地图保存为 `raw/_curation-maps/YYYY-MM-DD-主题.md`

## 七、覆盖报告（第 10 节）格式规范

覆盖报告是策展地图的最后一节。其生成方式和可信度取决于提取模式。

### 7.1 元信息

| 字段 | 类型 | 说明 |
|------|------|------|
| `生成模式` | enum | `独立审计 — Mode A` / `自检 — Mode A-fast` / `交叉校验 — Mode B/C` |
| `审查 Agent` | string | `独立 auditor` / `提取 Agent 同一调用` / `assembler` |
| `可信度说明` | string | 见下方说明 |

可信度说明：
- **独立审计** — 审查 Agent 与提取 Agent 分离，对照源文独立检查覆盖。可信度高。
- **自检** — 提取与覆盖报告同一次 LLM 产出。标注「非独立审计」。LLM 倾向于不标记自己的盲区——所以标记了异常更可能是真的。
- **交叉校验** — assembler 对照独立 Pass 的提取输出生成。可信度高。

### 7.2 节级覆盖

对照源文的自然段落/小节结构，标注每节的提取密度。Mode A / A-fast 额外包含「全景概括提及」列。

| 字段 | 类型 | 说明 |
|------|------|------|
| `源文节` | string | 节号或段落范围 |
| `原文首句/标题` | string | 该节起始文字或标题 |
| `全景概括提及` | enum | ☑ / ☒ / —。Mode A/A-fast 必填。Mode B/C 填 `—` |
| `提取项数` | number | 该节的提取项数量 |
| `覆盖状态` | enum | `☑ 正常` / `☑ 密集` / `⚠ 偏低` / `⚠ 盲区` |
| `说明` | string | 简短说明。盲区必须注明原因（如「代码示例」「参考文献」） |

覆盖统计：源文共 N 个大节，提取覆盖 M 个，K 个标记为盲区。总提取项数 X。

### 7.3 全景概括对照（仅 Mode A / A-fast）

全景概括逐段描述 vs 实际提取覆盖的差异。全景概括与提取来自同一 LLM 调用，但属于「提取前」和「提取后」两次注意力遍历——使用此差异作为有限交叉校验。

- 第 N 节概括提到「X、Y、Z」→ 提取项中缺少「Z」→ ⚠ 交叉校验异常

Mode B/C 不需要此节——已有独立 Pass 交叉校验。

### 7.4 密度异常

- 均值 = 总提取项数 / 覆盖的节数（排除盲区）
- 某节 > 3×均值 → 「密集」
- 某节 < 0.3×均值 且非盲区 → 「偏低」

### 7.5 结构一致性

源文共 N 个大节，覆盖 M 个，K 个盲区。覆盖率 = M / N。覆盖率 < 80% → ⚠ 异常。

### 7.6 盲区声明

| 字段 | 类型 | 说明 |
|------|------|------|
| `位置` | string | 源文段落/节位置 |
| `原因` | string | 盲区原因 |
| `内容类型` | string | 「代码块」「引用」「文字描述」等 |
| `C6 相关？` | enum | `是`（超长未细读）/ `否`（已读但判断不提取）。C6 盲区和本表盲区独立列出 |

如果所有盲区均为结构性内容（代码块、附录、参考文献），标注「均为结构性内容」。如果盲区包含文字描述段，标注 ⚠ 异常。

### 7.7 覆盖报告生成者职责

**Mode A auditor**（独立审计，模板 `sop-001-modeA-auditor.md`）：
- 输入：源文全文 + 策展地图第 1-9 节
- 任务：逐节检查提取项对源文的覆盖情况
- 限制：不做语义判断——不评估提取质量、不修正错误定义、不补充遗漏内容
- 模型：推荐便宜模型（Haiku / GPT-4o-mini 级别）

**Mode A / A-fast 提取 Agent**（自检，模板 `sop-001-modeA-fast.md`）：
- 全景概括优先于逐段提取
- 覆盖报告对照全景概括产出
- 标注为「自检——非独立审计」

**Mode B/C assembler**（交叉校验，模板 `sop-001-modeBC-assembler.md`）：
- 对照独立 Pass 的提取输出
- Mode B：不重读源文，仅基于 Pass 输出交叉对比
- Mode C：对照源文重读，含缺口分析和 SCHEMA 建议
